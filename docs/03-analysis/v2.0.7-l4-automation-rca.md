## §1. Q1 — Gemini CLI hook 'allow' decision 인식 (입증)

### 1.1 직접 증거 (bkit 자체 코드)

bkit `hooks/scripts/before-tool.js` 정독 결과 (`before-tool.js:30~98`):

| Line | Decision 반환 | 조건 |
|------|--------------|------|
| 58 | `{ decision: 'deny', reason: '...' }` | Permission deny |
| 68 | `{ decision: 'ask', systemMessage: '...' }` | Permission ask (PR #21146 v0.36.0+) |
| 86 | `{ decision: 'deny', reason: bashResult.message }` | Destructive bash 차단 |
| 92 | `{ decision: 'allow', systemMessage: contexts.join('\n') }` | PDCA warning 포함 allow |
| 94 | `{ decision: 'allow' }` | 정상 allow |
| 96 | `{ decision: 'allow' }` | catch 기본 allow (graceful) |

→ bkit이 v2.0.0 release 이후 production에서 이 4가지 decision을 정상 사용 중이고 회귀 0건. **Gemini CLI는 'allow' decision을 정상 인식한다**.

### 1.2 보조 증거 (PR #21146 인용)

`before-tool.js:63` 주석: `v0.36.0+: BeforeTool Hook supports 'ask' decision natively (PR #21146)`

- `lib/gemini/version.js`의 `hasBeforeToolAsk` capability flag (line 180): `isVersionAtLeast('0.36.0')`
- v0.36.0+ 환경(현 v0.42.0)에서 'ask'/'allow'/'deny' 3-state 정식 지원
- PR #21146 자체는 'ask' 추가 PR이지만, 그 전에 'allow'/'deny'는 이미 v0.34.0~ 지원되어 왔음

### 1.3 공식 문서 인용 (Gemini CLI docs)

[Gemini CLI Hooks reference](https://geminicli.com/docs/hooks/reference/):
- BeforeTool hook decision: `allow` | `deny` | `ask_user`
- `decision: 'allow'` 결정은 tool 실행을 정상 진행시킴

### 1.4 결론

> **Q1 답: Yes (확정)**. R7-G "allow silent ignore" 위험은 **사실상 해소**. Master Plan §1.3 R7-G 행을 "downgrade-from-high-to-low" 처리 권장.

## §2. Q2 — Track A 단독 L4 자동화 가능성

### 2.1 Track A 작동 메커니즘 재확인

S7 Design §5.1 `evaluateAutomation()` 함수가 다음 단계로 작동:
1. Level 확인 (L0~L4)
2. cmd-parser AST destructive 검사
3. trust-score 임계치 검사
4. tool whitelist 매칭
5. → `{ decision: 'allow' | 'deny' | 'ask' }` 반환

위 Q1 확인으로 'allow' 반환이 Gemini CLI에서 정상 인식됨이 입증. 따라서 Track A 단독으로 L4 자동화 작동이 가능.

### 2.2 잔여 위험 — Interactive mode (CC 이슈 #52822 유사)

[Anthropic CC issue #52822](https://github.com/anthropics/claude-code/issues/52822) (v2.1.119):
> "PreToolUse hook permissionDecision 'allow' does not suppress native permission prompt in interactive mode"

→ CC에서도 interactive mode에서 hook의 'allow' decision이 native permission prompt를 못 막는 사례 보고됨. **Gemini CLI에서 동일 이슈가 있다면 L4 자동화가 interactive mode (사용자가 terminal에서 직접 사용)에서는 부분 작동**.

**확인 절차 (Wave 0 Day 1 후속)**:
1. `tests/fixtures/gemini-cli-allow-suppress/before-tool.js`:
   ```js
   #!/usr/bin/env node
   console.log(JSON.stringify({ decision: 'allow', reason: 'auto-test' }));
   ```
2. `gemini -p "ls"` (interactive 가까운 mode)
3. Gemini CLI가 native prompt 표시하는지 / suppress하는지 관찰
4. 결과를 RCA §2.4에 기록

**Fallback 전략** (만약 interactive mode에서 'allow'가 silent suppress 안 되는 경우):
- L4 모드는 non-interactive (--prompt) 호출에서만 작동 보장
- Interactive mode는 L3까지 (사용자가 명시적 '/control level L3' 후 prompt 응답)
- 또는 Track B PR로 interactive mode 'allow' 동작 명확화 (sandbox 채널 추가)

### 2.3 결론

> **Q2 답: Yes (높은 확신)** — non-interactive (--prompt) 모드. Interactive mode는 Wave 0 Day 1 reproducer로 확정.

### 2.4 Wave 0 Day 1 Reproducer 결과 (2026-05-14)

**Case A — Non-interactive 'allow' suppress (확정 PASS)**:

```
$ GEMINI_CLI_TRUST_WORKSPACE=true gemini -p "Respond with exactly: OK_RCA_DAY1"

bkit Vibecoding Kit v2.0.7 activated (Gemini CLI) - Level: Starter
Hook system message: bkit Vibecoding Kit v2.0.7 activated (Gemini CLI) - Level: Starter
OK_RCA_DAY1
Hook system message: bkit session ended. State preserved.
```

**관찰**:
- ✅ bkit hooks 정상 활성 (SessionStart + before-tool + SessionEnd)
- ✅ LLM 응답 정상 (OK_RCA_DAY1)
- ✅ **Prompt 0건** — bkit `before-tool.js`의 'allow' decision이 Gemini CLI에서 정상 suppress
- ✅ Tool 실행 (Bash `gemini -p` invocation 자체) prompt 없이 진행

**Q2 Case A 답 (확정)**: ✅ **Non-interactive 모드에서 'allow' decision이 native prompt를 완전 suppress**. Track A의 L4 자동화 메커니즘이 production에서 작동 가능함이 직접 입증됨.

**Case B — Interactive mode** (사용자 별도 검증 권장):
- Bash 도구로 자동화 불가 (TTY 입력 필요)
- 사용자가 별도로 다음 시나리오 검증 권장 (Day 2 Wave 1 진행과 무관, 정보성):
  1. `gemini` 시작 (interactive)
  2. prompt에서 `ls` 입력
  3. bkit before-tool 'allow' 결정 시 native prompt 표시 여부 관찰
- **만약 interactive에서 prompt 표시되면**: Track B PR-1 (interactive mode 'allow' fix) 우선순위 ↑
- **만약 정상 suppress되면**: Track B PR 자체 불필요, Final Report에 명시

**Wave 0 Day 1 종료 판정**: ✅ PASS — Q1/Q2(Case A)/Q3/Q4 모두 입증/결정 완료. Day 2 Wave 1 진입 가능.

## §4. Q4 — D16 AST 라이브러리 재조정

### 4.1 bash-parser (vorpaljs/bash-parser, npm) 실측

| 항목 | 값 |
|------|---|
| 최신 버전 | 0.5.0 |
| 마지막 publish | **9년 전 (~2017)** |
| 유지보수 상태 | **사실상 abandoned** |
| 다운로드 | 낮음 |
| 의존 패키지 수 | 22 |
| License | MIT |
| heredoc 지원 | 부분 (BUG 보고 있음) |

**결론**: 사용자의 production 안전 요구 + "꼼꼼하게" 원칙과 부합 안 함. **reject**.

### 4.2 대안 평가

| 후보 | License | Maintenance | Heredoc | Cross-OS | 결정 |
|------|---------|-------------|---------|----------|------|
| `bash-parser` (vorpaljs) | MIT | abandoned (2017) | 부분 | ✅ | **reject** |
| `@ericcornelissen/bash-parser` (fork) | MIT (추정) | 활성 추정 | 부분 | ✅ | **검토** |
| `tree-sitter-bash` (Microsoft) | MIT | 활성 (LSP) | ✅ | ✅ (WASM) | **검토** (의존성 큼) |
| `mvdan/sh` (Go binary) | BSD-3 | 활성 | ✅ | ⚠ Windows | **reject** (이전 결정) |
| **자체 mini-parser only** | bkit (Apache) | bkit 자체 | 자체 구현 | ✅ | **유력** |

### 4.3 추천 D16 (revised)

**자체 mini-parser only + optional `@ericcornelissen/bash-parser` 2차 검사**:

- **1차 — 자체 mini-parser** (`lib/core/cmd-parser.js`, ~250L):
  - regex + token 분석
  - heredoc, subshell, IFS, eval, exec, xargs, fd redirection, ANSI escape, alias override 11종 bypass 검출
  - 16 destructive 패턴 (D-RM-RF~D-DROP-DATABASE)
  - 자체 unit test 31 케이스 (CP-01~31)
  - **의존성 0, cross-OS 100%, bkit 직접 maintain**
- **2차 — `@ericcornelissen/bash-parser` (선택, defer to v2.1.0)**:
  - 자체 mini-parser가 의심하는 명령에만 적용
  - v2.0.7 S7 scope에서는 **out-of-scope** (자체 parser로 충분)
  - v2.1.0에서 정밀 검증 도입 시 재평가

### 4.4 결정 근거

1. **maintenance 신뢰성**: 자체 코드는 bkit이 직접 fix 가능
2. **Cross-OS 100%**: pure JavaScript regex, OS 의존성 0
3. **scope 통제**: v2.0.7 S7은 prerequisite sprint, 의존성 추가는 v2.1.0+ 시점이 적절
4. **사용자 명시 원칙** "꼼꼼하게": 외부 abandoned 라이브러리 의존은 위험. 자체 코드로 31 케이스 회귀 testcase 보장

### 4.5 결론

> **Q4 답: D16 revised = 자체 mini-parser only**. PRD §8의 `bash-parser npm 의존성` 행 제거. Plan §6.1 D16 갱신 필요.

→ D5 (read-only whitelist)는 이미 정의되어 있음. before-tool.js에 간단히 import + 적용하면 됨.

- → D10 (audit log)의 base는 이미 구축. v2.0.7 S7에서는 ALLOW 이벤트 추가 + JSONL 마이그레이션만 필요.
