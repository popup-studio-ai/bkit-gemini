## 1. Problem Definition

### 1.1 현재 상태

| 카테고리 | 위치 | 실 LLM 호출 여부 | PR-time CI 포함 여부 |
|---------|------|----------------|---------------------|
| L1 Unit (`tests/suites/*.js`, `tests/regression/**/*.test.js`) | 자동 | ❌ 없음 (mock/grep/static) | ✅ 포함 |
| L2 Integration | `tests/run-all.js --strict` | ❌ 없음 | ✅ 포함 |
| L3 Spot (`gemini -p`) | `docs/test-plans/v0.42.0-bkit-extension-test-plan.md §4` | ✅ **실 LLM 호출** | ❌ 미포함 (manual) |
| L4 Agent dispatch | 동 §5 | ✅ 실 LLM 호출 | ❌ 미포함 (manual) |
| L5 E2E canary (10건 시나리오 A/B1/B2/C/E1~E6) | 동 §6 | ✅ 실 LLM 호출 | ❌ 미포함 (manual) |

### 1.2 핵심 사실

**`tests/run-all.js --strict`는 이미 LLM 호출 0회**. 1956/2051 strict 결과는 모두 정적/단위/통합 검증. 따라서 strict 자체는 "stub 모드 호환".

문제는 **L3/L4/L5 manual scenario test 10여 건이 자동화되지 않아 PR-time CI에 포함되지 못함**. 자동화하려면 LLM 호출 비용 + 비결정성 + 시간 문제.

### 1.3 진짜 필요한 것

- L3/L4/L5 시나리오를 **자동화** 가능한 framework
- 자동화 시 LLM 호출을 **record/replay**로 결정성 확보
- `BKIT_TEST_MODE=stub` 로 PR-time CI에서 LLM 호출 0회 + 5분 내 완료
- `BKIT_TEST_MODE=record` 로 nightly 또는 manual capture 시 fixture 자동 update
- `BKIT_TEST_MODE=live` (기본) 로 기존 동작 보존

## 2. Architecture

### 2.1 Components

```
┌────────────────────────────────────────────────────────────────┐
│  Scenario test (auto)                                           │
│    e.g. tests/scenarios/L3-1-onetwothree.test.js                │
│      ↓ calls                                                    │
│  lib/test/llm-replay.js  (NEW)                                  │
│    recordOrReplay(request) {                                    │
│      switch (BKIT_TEST_MODE)                                    │
│        'stub'   → fixture lookup or fail-fast                   │
│        'record' → live spawn + write fixture                    │
│        'live'   → live spawn only                               │
│    }                                                            │
│      ↓ spawn (live/record mode)                                 │
│  gemini (real CLI)                                              │
│      ↓ response                                                 │
│  tests/fixtures/llm-responses/L3-1-onetwothree.json (NEW)       │
│    { request_hash, request, response, captured_at, version }    │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 BKIT_TEST_MODE state matrix

| Mode | LLM 호출 | Fixture 읽기 | Fixture 쓰기 | Hash miss 시 |
|------|---------|------------|-------------|--------------|
| `stub` (default in CI) | ❌ | ✅ | ❌ | fail-fast with hint |
| `record` (manual / nightly capture) | ✅ | ❌ | ✅ | fixture 자동 추가 |
| `live` (default in dev / nightly verify) | ✅ | ❌ | ❌ | 정상 동작 |

### 2.3 Request hash 정책

```js
hash = sha256(
  JSON.stringify({
    args:        normalized CLI args (sorted, prompt 포함),
    env:         { GEMINI_CLI_TRUST_WORKSPACE, GEMINI_CLI_VERSION,
                   BKIT_SESSION_START_VERBOSE, ... allowlisted only },
    cwd_relative: cwd basename (절대경로 누설 X),
    bkit_version: package.json version
  })
).slice(0, 16);
```

`captured_at` + `gemini_cli_version`은 hash 외 메타. drift 검사용.

## 5. Risks

| Risk | Mitigation |
|------|-----------|
| Fixture 캡처 시점과 실제 LLM drift → false-pass | hash + captured_at + version 메타 기록. nightly drift 검증은 별도 sprint. |
| Hash 입력 불안정 (env var ordering / cwd absolute path) | hash 함수가 정렬 + 상대경로화. 동적 회귀 test로 검증. |
| stub mode에서 hash miss 시 silent pass | fail-fast 패턴 — Error throw + hint message. 동적 회귀 검증. |
| LLM record mode 운영 시 비용 발생 | record는 manual 또는 nightly only. 기본 mode는 live (dev) / stub (CI). |

## 6. 임시방편 vs 근본 대응

| 임시방편 (NO) | 근본 대응 (YES) |
|---------------|----------------|
| L3/L4/L5 시나리오를 PR-time CI에서 그냥 skip | record/replay framework로 자동화 + LLM 호출 0회 보장 |
| fixture를 manual JSON으로 합성 | record mode로 실 LLM 응답 캡처 → version + captured_at 추적 |
| hash miss 시 silently fallback to live | fail-fast + hint message → CI에서 즉시 확인 가능 |
| BKIT_TEST_MODE를 무시하고 모든 환경에서 live | env var 명시 분기 + 동적 회귀로 mode별 동작 강제 검증 |
