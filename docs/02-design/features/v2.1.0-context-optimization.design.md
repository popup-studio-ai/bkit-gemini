## 1. Design Overview

### 1.1 변경 범위 요약

| Category | Action | Count | Token Impact |
|----------|--------|-------|-------------|
| **Module Connection** | Import 누락 모듈을 hooks/MCP에 연결 | 7 modules | Functionality gain |
| **PM Bug Fix** | 4 PM agents를 AGENTS registry에 등록 | 4 agents | +~330 tokens/turn |
| **Context Trim** | 중복 context 제거, @import 최적화 | 5 items | **-2,143 tokens/turn** |
| **Dead Code Delete** | 진짜 중복 모듈 + orphan hooks 삭제 | 8 files | Code cleanliness |
| **Hook Optimization** | SessionStart→BeforeAgent context 이동 | 2 hooks | **-900 tokens residual** |
| **Hook Utility** | 중복 코드 통합 유틸리티 | 1 new file | Maintainability |
| **Net Token Savings** | | | **~2,700 tokens/turn (-27%)** |

### 1.2 Architecture (변경 없음 — 3-Layer 유지)

```
┌─────────────────────────────────────────────────┐
│  Gemini CLI                                      │
│                                                   │

│       ▲ 모듈 연결 보완       ├ bkit_pm_run        │
│       │                      ├ bkit_qa_run        │
```

### 1.3 3 Architecture Options

| Option | Description | Effort | Risk |
|--------|-------------|--------|------|
| A. Context Only | @import 정리 + 중복 제거만 | 8h | Very Low |
| B. Context + Connect | A + 7개 모듈 연결 | 20h | Low |
| **C. Full Slim ★** | B + orphan 삭제 + hook 최적화 + skill 정리 | **30h** | **Low** |

**Option C 선택 이유**: 모듈 연결은 기능 완성도를 높이고, context 정리는 성능을 개선하며, 둘 다 해야 bkit의 의도가 완전히 실현됨.

## 2. Module Connection Specification (CONNECT 7개)

### 2.1 lib/core/config.js → session-start.js

**의도**: bkit.config.json을 TTL 캐시로 로드하여 반복 파일 읽기 방지

**Before** (session-start.js 현재):
```javascript
// 매 세션마다 직접 파일 읽기
function loadBkitConfig(extensionPath) {
  try {
    const configPath = path.join(extensionPath, 'bkit.config.json');
    return JSON.parse(fs.readFileSync(configPath, 'utf-8'));
  } catch { return {}; }
}
```

**After**:
```javascript
const { loadConfig } = require(path.join(libPath, 'core', 'config'));

// TTL 캐시 활용 — 같은 세션 내 반복 호출 시 파일 I/O 없음
const bkitConfig = loadConfig(extensionPath);
```

**영향**: session-start.js의 loadBkitConfig() 인라인 함수 제거. lib/core/config.js 사용.
**성능**: 파일 I/O 1회 → 캐시 히트 (같은 프로세스 내 재호출 시)

### 2.2 lib/core/agent-memory.js → after-agent.js

**의도**: 에이전트 실행 후 학습 데이터를 세션 간 유지 (FR-08 State Management)

**Before** (after-agent.js 현재):
```javascript
// 에이전트 결과만 PDCA status에 기록, 에이전트별 메모리 없음
```

**After**:
```javascript
const { getAgentMemory } = require(path.join(libPath, 'core', 'agent-memory'));

// 에이전트 완료 후 메모리에 결과 기록
try {
  const agentMem = getAgentMemory(agentName, projectDir);
  agentMem.recordSession({
    phase: pdcaStatus?.phase,
    result: result?.summary || 'completed',
    timestamp: new Date().toISOString()
  });
} catch (_) { /* best-effort */ }
```

**영향**: 에이전트가 이전 세션의 패턴/결과를 참조하여 더 나은 분석 가능.
**[HALLUCINATION PREVENTION]**: 이전 결과를 기억하므로 같은 실수 반복 방지.

### 2.3 lib/core/debug.js → session-start.js, before-tool.js

**의도**: 디버그 로그를 stdout (JSON 출력)과 분리. Hook JSON 파싱 실패 방지.

**Before**:
```javascript
// console.error로 디버그 — stderr과 혼재
console.error('[bkit] debug:', someInfo);
```

**After**:
```javascript
const { debugLog } = require(path.join(libPath, 'core', 'debug'));

// ~/.gemini/bkit-debug.log에 별도 기록
debugLog('session', 'Session initialized', { level, feature });
```

**영향**: BKIT_DEBUG=true 시에만 활성화. stdout 오염 제거로 hook 안정성 향상.

### 2.4 lib/core/file.js → before-tool.js, after-tool.js

**의도**: 파일 타입 감지를 중앙화 (source/UI/env/tier)

**Before** (before-tool.js 현재):
```javascript
// 하드코딩된 확장자 패턴
const sensitivePatterns = ['.env', '.pem', '.key', '.cert'];
if (sensitivePatterns.some(p => filePath.includes(p))) { ... }
```

**After**:
```javascript
const { isEnvFile, isSourceFile, getLanguageTier } = require(path.join(libPath, 'core', 'file'));

if (isEnvFile(filePath)) { ... }  // .env, .env.local, .env.production 등 포괄
```

**영향**: 일관된 파일 분류. 새 확장자 추가 시 한 곳만 수정.

### 2.5 lib/gemini/context-fork.js → mcp/bkit-server.js (bkit_iterate)

**의도**: Gap 분석 시 메인 PDCA 상태를 보호 (FR-03 No Guessing)

**Before** (bkit_iterate 현재):
```javascript
// 분석이 메인 상태에 직접 접근
const result = await gapAnalyzer.analyze({ feature, projectDir, designPath });
```

**After**:
```javascript
const { forkContext, mergeForkedContext } = require(path.join(extensionPath, 'lib', 'gemini', 'context-fork'));

// 스냅샷 → 분석 → 결과만 머지
const snapshot = forkContext(projectDir, `iterate-${feature}`);
try {
  const result = await gapAnalyzer.analyze({ feature, projectDir, designPath });
  mergeForkedContext(snapshot, { matchRate: result.matchRate });
  return result;
} catch (e) {
  // 분석 실패 시 스냅샷 폐기, 메인 상태 무영향
  return { error: e.message };
}
```

**[HALLUCINATION PREVENTION]**: 분석 중 상태 변경이 메인에 영향 안 줌.

### 2.6 lib/pdca/level.js → session-start.js

**의도**: 프로젝트 레벨 감지를 중앙화 (Starter/Dynamic/Enterprise)

**Before** (session-start.js 현재):
```javascript
// 인라인 detectProjectLevel() — ~50줄 하드코딩
function detectProjectLevel(projectDir) {
  if (fs.existsSync(path.join(projectDir, 'kubernetes'))) return 'Enterprise';
  if (fs.existsSync(path.join(projectDir, 'terraform'))) return 'Enterprise';
  // ... 20+ more checks
}
```

**After**:
```javascript
const { detectLevel } = require(path.join(libPath, 'pdca', 'level'));

const level = detectLevel(projectDir);
// detectLevel()은 canSkipPhase(), getRequiredPhases() 도 제공
```

**영향**: 인라인 ~50줄 제거. 레벨별 스킵 가능 phase 판단 기능 추가.

### 2.7 lib/pdca/phase.js → mcp/bkit-server.js, after-agent.js

**의도**: PDCA 단계 전환을 검증하여 역방향 이동 방지

**Before**:
```javascript
// 인라인 전환 — 검증 없음
pdcaStatus.phase = 'design';  // plan → design은 OK지만 design → plan도 통과
```

**After**:
```javascript
const { validatePdcaTransition, getNextPdcaPhase } = require(path.join(extensionPath, 'lib', 'pdca', 'phase'));

const validation = validatePdcaTransition(currentPhase, targetPhase);
if (!validation.valid) {
  return { error: `Invalid transition: ${validation.reason}` };
}
```

**[HALLUCINATION PREVENTION]**: 잘못된 단계 전환 방지 (예: check에서 plan으로 되돌아가기 차단).

## 3. Context Optimization Specification

### 3.1 GEMINI.md @import 최적화 [PERFORMANCE]

**Gemini CLI 동작**: GEMINI.md의 모든 @import는 매 턴 인라인됨 (eager, 전체 토큰 비용)

**Before** (현재):
```markdown
@.gemini/context/commands.md     ← 566 tokens, 매 턴
@.gemini/context/core-rules.md   ← 385 tokens, 매 턴
```

**After**:
```markdown
@.gemini/context/core-rules.md   ← 385 tokens, 매 턴 (PDCA 규칙 — 할루시네이션 방지 필수)
```

**commands.md 처리**: 
- 삭제하지 않음 — `/help` 명령어 참조용으로 유지
- @import에서만 제거 — 사용자가 명령어를 물어볼 때 에이전트가 직접 읽기
- **Savings: -566 tokens/turn**

**[HALLUCINATION PREVENTION]**: core-rules.md는 유지 — PDCA 워크플로우 규칙, 문서 우선 원칙, 추측 금지 규칙 포함.

### 3.2 중복 PDCA Rules 통합 [PERFORMANCE]

3곳에 동일 내용 존재:
1. `.gemini/context/core-rules.md` (385 tokens) — @import, 매 턴 ✅ 유지
2. `.gemini/context/pdca-rules.md` (353 tokens) — hook conditional → **삭제, core-rules.md에 통합**
3. `session-start.js buildCoreRules()` (~300 tokens) — hook 주입 → **제거**

**After**: core-rules.md 하나로 통합. PDCA 규칙은 @import로 매 턴 로드되므로 hook 재빌드 불필요.
**Savings: -~580 tokens**

### 3.3 Trigger 문서 파일 이동 [PERFORMANCE]

`.gemini/context/agent-triggers.md` (788 tokens) + `skill-triggers.md` (912 tokens):
- lib/intent가 동적으로 intent matching을 수행하므로 이 정적 문서는 **코드에서 미사용**
- Hook이 conditional로 로드하더라도 토큰 낭비

**Action**: `docs/reference/`로 이동 (context에서 제거)
**Savings: -1,700 tokens** (hook이 로드하는 경우)

### 3.4 SessionStart → BeforeAgent Context 이동 [PERFORMANCE]

**Gemini CLI 동작**:
- SessionStart additionalContext → 대화 히스토리에 **잔존** (compression까지)
- BeforeAgent additionalContext → **이번 턴만** (가장 효율적)

**Before**: session-start.js가 phase context, level info, style info를 additionalContext로 주입 → 히스토리에 ~1,500 tokens 잔존

**After**: 
- session-start.js → 최소한의 초기화만 (PDCA status load, level detect, memory init)
- before-agent.js → 턴별 phase context, 가이드를 additionalContext로 주입
- **Savings: -~900 tokens residual** in conversation history

### 3.5 Agent Description 압축 [PERFORMANCE]

21 agents × ~83 tokens = 1,750 tokens/turn

대부분의 agent description에 8개국어 트리거 + 상세 설명이 포함됨.
Gemini CLI는 description 전체를 system prompt에 넣으므로, 압축 필요.

**Target**: 평균 60 tokens (from 83)
**Method**: 상세 설명 축약, 트리거는 EN/KO/JA/ZH 4개 유지 (ES/FR/DE/IT 제거 — 사용률 <1%)
**Savings: -~480 tokens/turn**

## 6. Hook Utility 통합

### 6.1 hooks/scripts/utils/hook-init.js (NEW)

10+ hook 스크립트에 반복되는 초기화 코드를 통합:

```javascript
/**
 * Common hook initialization utility
 * Eliminates 10+ duplicated init patterns across hook scripts
 */
'use strict';
const path = require('path');
const fs = require('fs');

const extensionPath = process.env.GEMINI_EXTENSION_PATH
  || process.env.extensionPath
  || path.resolve(__dirname, '..', '..', '..');

const libPath = path.join(extensionPath, 'lib');
const projectDir = process.env.GEMINI_PROJECT_DIR || process.cwd();

// Lazy-loaded modules (only require when needed)
let _adapter, _pdcaStatus, _config;

function getAdapter() {
  if (!_adapter) {
    const { getAdapter: ga } = require(path.join(libPath, 'gemini', 'platform'));
    _adapter = ga();
  }
  return _adapter;
}

function getPdcaStatus() {
  if (!_pdcaStatus) {
    const { loadPdcaStatus } = require(path.join(libPath, 'pdca', 'status'));
    _pdcaStatus = loadPdcaStatus(projectDir);
  }
  return _pdcaStatus;
}

function getConfig() {
  if (!_config) {
    const { loadConfig } = require(path.join(libPath, 'core', 'config'));
    _config = loadConfig(extensionPath);
  }
  return _config;
}

function readHookInput() {
  try {
    const input = fs.readFileSync(0, 'utf-8').trim();
    return input ? JSON.parse(input) : {};
  } catch { return {}; }
}

function outputAllow(systemMessage) {
  const output = { decision: 'allow' };
  if (systemMessage) output.systemMessage = systemMessage;
  console.log(JSON.stringify(output));
  process.exit(0);
}

function outputDeny(reason) {
  console.log(JSON.stringify({ decision: 'deny', reason }));
  process.exit(2);
}

module.exports = {
  extensionPath, libPath, projectDir,
  getAdapter, getPdcaStatus, getConfig,
  readHookInput, outputAllow, outputDeny
};
```

**효과**: 각 hook 스크립트에서 ~15줄 초기화 코드 → `require('./utils/hook-init')` 1줄로.
