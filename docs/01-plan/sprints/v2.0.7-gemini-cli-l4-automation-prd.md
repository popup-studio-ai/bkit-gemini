## §2. Job Stories (Beachhead → Mainstream)

### 2.1 Beachhead User: @kay (CTO, bkit-gemini owner, v2.0.7 Debt Resolution 진행자)

**Beachhead Job Story #1 — Sprint 시작 정책 자동화**

> When 나는 v2.0.7 Debt Resolution Program의 S1~S6 sprint를 차례로 진행할 때,
> I want to 한 sprint당 한 번의 `/sprint start <name>` + `/control level L4` 명령으로 PDCA 6 phase가 사용자 결재 없이 진행되기를 원한다,
> So I can 임시방편 거부 + 품질·성능·완성도 최대화 원칙을 지키면서 wall-clock time을 50% 단축할 수 있다.

→ **AC-T1, AC-T2 매핑**

**Beachhead Job Story #2 — destructive 영구 차단 신뢰**

> When 나는 L4 자동 모드에서 sprint를 실행할 때,
> I want to `rm -rf`, `git push --force`, `drop table`, `truncate`, `DELETE FROM`, `sudo`, `chmod 777`, `/etc/*` 수정, `~/.ssh/*` 수정이 trust_score / level과 무관하게 영구 deny 되기를 원한다,
> So I can L4 자동화를 신뢰하면서도 production 데이터/시스템 손상 0건 보장을 받을 수 있다.

→ **AC-T3, AC-T9 매핑**

**Beachhead Job Story #3 — 의도치 않은 변경 즉시 되돌리기**

> When 나는 L4 모드 진행 중 결과가 잘못되었다고 판단할 때,
> I want to `/control rollback` 한 번으로 마지막 N 결정을 사용자 결재 없이 되돌리기를 원한다 (왜냐하면 결재 받으려면 또 L2로 강등되어야 하는데 그게 다음 결재 cycle을 만든다),
> So I can 자동화의 속도를 잃지 않고 안전망만 확보할 수 있다.

→ **AC-T5 매핑**

### 2.2 Mainstream User: bkit Gemini CLI 사용자 일반 (v2.1.0+ 대상)

**Mainstream Job Story #4 — 21 agent 무인 spawn**

> When 나는 PDCA Plan phase에서 code-analyzer agent에게 분석을 맡길 때,
> I want to L4 자동 모드에서 agent spawn 결재를 매번 받지 않기를 원한다 (단, agent 결과의 destructive write는 여전히 영구 deny),
> So I can sprint-orchestrator가 21 agent 협업을 자동으로 진행할 수 있다.

→ **AC-T2 (phase 전환의 부속), AC-T8 (audit log) 매핑**

**Mainstream Job Story #5 — 신뢰 자동 회복**

> When 나는 L4 모드에서 잘못된 결정 5건이 누적되어 자동 L2 강등 + 24h cooldown을 경험한 후,
> I want to 24h 후에 cooldown 자동 해제 + trust_score 점진 회복 (success +5/decision)으로 다시 L4로 진입할 수 있기를 원한다,
> So I can 시스템이 한 번의 실수로 영구 강등되지 않는다.

→ **AC-T4, AC-T13 매핑**

### 2.3 Pre-mortem Job Story (Anti-Job)

**Anti-Job #1 — L4 만능키 만들기 거부**

> When 누군가가 "L4 모드에서는 모든 tool을 무조건 allow하자" 패턴을 제안할 때,
> I want to destructive 매트릭스가 trust_score / level과 독립적으로 영구 deny되기를 원한다 (level과 무관하게 hard deny),
> So I can L4 자동화가 "사용자 결재 없는 destructive 폭주"로 변질되지 않는다.

→ **AC-T3, AC-T9 매핑**

## §4. 기능 명세 (요약, Plan/Design에서 상세화)

### 4.1 trust_score 알고리즘

```
초기값: 40 (현재)
성공 결정 (사용자 unrejected): trustScore += 5 (cap 100)
거절 결정 (사용자 reject 또는 hook deny): trustScore -= 10 (cap 0)
5건 연속 rejection: trustScore = 30 (force) + level L2 강등 + cooldownUntil = now + 24h
L4 entry 조건: trustScore >= 80 AND cooldownUntil < now AND 최근 10 결정 중 rejection == 0
oscillation 방지: level 변경 후 1h 내 재변경 deny (100ms boundary 잡음 무시는 별도 debounce)
```

### 4.2 L4 안전 가드 8-Layer (Design §4 상세)

| Layer | 가드 | 작동 시점 |
|-------|------|----------|
| L1 | Hard deny matrix (destructive 영구) | before-tool hook |
| L2 | trust_score gate (>= 80 + cooldown + 최근 10건 clean) | before-tool / before-phase hook |
| L3 | Auto checkpoint (write 전 `.bkit/checkpoints/{ts}-{op}.json`) | after-tool hook |
| L4 | trust_score auto-downgrade (5 rejection 연속) | after-tool hook |
| L5 | Emergency stop (`/control stop` 또는 Ctrl+C 2회) | session-end / SIGINT handler |
| L6 | Audit log (모든 L4 결정 영구 기록) | after-tool hook |
| L7 | SessionEnd review (의문 결정 사용자 검토 prompt) | session-end hook |

### 4.3 destructive 매트릭스 (5-layer 정규식)

```javascript
// Layer 1: bash command direct
/^(rm\s+-rf\s+\/|sudo\s|chmod\s+777|>\s*\/etc\/|>\s*~\/\.ssh\/)/

// Layer 2: heredoc bypass detection
/<<\s*EOF[^]*?(rm\s+-rf|drop\s+table|truncate)[^]*?EOF/i

// Layer 3: SQL destructive
/\b(drop\s+(table|database|schema)|truncate\s+table|delete\s+from\s+[^\s;]+\s*(;|$))/i

// Layer 4: subshell / pipe escape
/\$\((rm\s+-rf|sudo|drop)/i

// Layer 5: AST-based command parsing (lib/core/trust-score.js의 parseShellAst())
// shellwords로 토큰화 후 첫 토큰이 destructive 화이트리스트와 매치
```

## §6. Pre-mortem (가상 실패 시나리오)

### 6.1 실패 시나리오 A: "L4 진입 후 첫 sprint에서 코드 의도치 않게 삭제"

**Causal chain**:
1. AC-T3 destructive 매트릭스에 누락 패턴 (예: `find . -delete`)
2. before-tool.js layer 1~5 모두 miss
3. write_file이 destructive 결과를 commit
4. 사용자 변경 손실

**예방**:
- AC-T3 10 시나리오에 `find . -delete` 포함 (Plan §3 destructive matrix 표에 명시)
- Layer 5 AST 파싱이 `find` 명령의 `-delete` flag 검출
- AC-T5 checkpoint가 write_file 전 자동 백업

**Detection**:
- AC-T8 audit log에 `tool: write_file, destructive_score: HIGH` 기록
- session-end review (L7)에서 사용자 검토 prompt

**Recovery**:
- `/control rollback` 단일 명령으로 무결재 복원
- trust_score force 0 + 24h cooldown

### 6.2 실패 시나리오 B: "trust_score boundary oscillation"

**Causal chain**:
1. trust_score = 80 (L4 진입 boundary)
2. 1 decision 실패 → 70 → L2 강등
3. 다음 4 decision 성공 → 80 → L4 재진입
4. 다음 1 decision 실패 → 70 → L2 강등
5. 사용자 경험: 깜빡임, 신뢰 침식

**예방**:
- AC-T4 + NFR-3: level 변경 후 1h 내 재변경 deny (cooldown debounce)
- 5건 연속 rejection 강한 임계값 (1건이 아닌 5건)

**Detection**:
- `automationHistory` 24h 내 2회 이상 level 변경 시 alert

**Recovery**:
- 사용자 수동 `/control level L2` 강제 (자동 L4 disable)

### 6.3 실패 시나리오 C: "Gemini CLI 'allow' 반환 silent ignore"

**Causal chain**:
1. F7-1 RCA 미수행 또는 reproducer 부정확
2. before-tool.js `{decision: 'allow'}` 반환
3. Gemini CLI는 'allow'를 무시하고 사용자에게 결재 prompt
4. AC-T2 실패 (phase 전환 무인 자동 불가)

**예방**:
- F7-1 RCA 단계에서 reproducer 작성 + 실제 동작 확정 (Phase 2 사용자 결재 gate)
- 'allow' silent ignore 확인 시 alternative path (sandbox-mode 환경변수 또는 force-permission 채널) 검토 + Plan §2.2 D1에 결정 사항 기록

**Mitigation (fallback)**:
- Track B PR fast-track (Gemini CLI 본체에서 'allow' 명시 인식 추가)
- 우회 path: BKIT_FORCE_LEVEL=L4 환경변수 (Plan §2.2 D2 결정 시)
