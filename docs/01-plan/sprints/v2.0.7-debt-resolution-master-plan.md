## §0. Executive Summary

### 0.1 왜 v2.0.7 안에서 P1~P6를 마저 끝내는가

v2.0.7-upgrade sprint는 Gemini CLI v0.42.0 stable과의 **호환성**을 100% 확보했지만, 검증 과정에서 발견된 3개 부채(R-extra-1 / R-extra-2 / C-future-1)와 2개 개선 영역(Manual LLM Verification / Baseline Stabilization)이 **다음 minor(v2.1.0)로 이월**된 상태이다. 이월은 일반적인 sprint 위생이지만, 본 program은 다음 이유로 v2.0.7 안에서 마저 해결한다:

1. **부채가 사용자 가치 직접 차단**: R-extra-1은 21개 specialized agent를 무력화하는 critical 부채. v2.1.0까지 미루면 "bkit이 광고하는 agent 가치 ≠ 실제 동작"이 1 minor cycle 더 지속.
2. **상호 의존**: C-future-1(Context Engineering)의 Sub-agent isolation은 R-extra-1(Dispatch) 위에서 동작. 별도 minor로 쪼개면 통합 비용 2배.
3. **테스트 인프라 회수**: CI Stub Mode(P3)와 Baseline Recovery(P1+P6)는 v2.1.0 이후 모든 sprint의 사전 인프라. v2.0.7에 누적해야 v2.1.0+ 속도 회복.

### 0.2 임시방편 명시적 거부

본 program은 임시 워크어라운드를 거부한다. 예를 들어 R-extra-1(Track A: bkit hook 우회)은 단독으로는 임시방편이지만, Track B(Gemini CLI 본체 PR)와 짝으로 진행하여 **근본 해결 시점까지 사용자 차단 0** + **본체 머지 후 우회 비활성화 옵션** 을 함께 제공한다. 모든 6 sprint는 동일 원칙 적용.

## §4. Sprint S3 — v2.0.7-agent-dispatch-fix (P4, Critical)

### 4.1 문제 정의

본 program의 **핵심 부채**. `gemini -p "Use the code-analyzer agent..."` → Gemini CLI 본체 `LocalSubagentInvocation.execute` 가 agent registry에서 `code-analyzer`를 404로 처리 → `generalist` fallback. 결과:

```
실제 사용자 흐름:
[사용자] "Use the security-architect agent for OWASP audit"
[Gemini] → "security-architect was not found, falling back to generalist"
[Gemini] → [generalist의 두루뭉술한 보안 조언]
```

광고된 21개 specialized agent(`code-analyzer`/`security-architect`/`gap-detector`/`pdca-iterator`/`qa-strategist` 등) 가치가 실현 불가. bkit이 광고하는 PDCA 깊이의 핵심 메커니즘이 손상된 상태.

근거 파일:
- `agents/*.md` 21개 — frontmatter 완전 (name/model/tools/temperature/max_turns 모두 정상)
- `mcp/bkit-server.js:321` — `spawn_agent` tool 이미 `enum: Object.keys(AGENTS)`로 21 agent 등록
- `mcp/bkit-server.js:319~400` — list_agents/get_agent_info/team_create/team_assign 완성
- Gemini CLI 본체 `LocalSubagentInvocation` — 외부 코드, agent registry 조회 로직 손상 추정

## §6. Sprint S5 — v2.0.7-context-engineering-integration (P2)

### 6.1 문제 정의

Anthropic 2025-09 발표 Context Engineering 4-strategy (Compaction / Tool-result clearing / Memory tool / Sub-agent isolation)가 bkit에 완전 통합되지 않은 상태. v2.0.7-upgrade ENH-3은 mapping 분석 문서(`docs/02-design/agent-memory-vs-extraction-patch-mapping.md`, ~600L)만 작성, **실제 통합 0건**.

현재 baseline:
- `lib/gemini/context-fork.js`: Context Fork 시뮬레이션, `jitPartial` 마커 (Compaction의 사전 작업)
- `lib/core/agent-memory.js` (214L): AgentMemoryManager class, project/user scope, 21 file 격리 ↔ Memory tool 통합의 baseline
- `hooks/scripts/session-start.js`: JIT dedup (GEMINI_MD_IMPORTS Set), Phase-Aware Context Loading
- `hooks/scripts/pre-compress.js`: 존재 확인 필요 (PreCompress hook의 trigger 시점)

사용자 영향: 긴 PDCA sprint(2~5시간) 진행 시 토큰 누적, 응답 지연, 컨텍스트 오염으로 인한 hallucination 가능성.

### 6.2 근본 대응 (S3 dispatch fix 후 진행 — Sub-agent isolation 의존)

4 strategy 단계 통합:

**Strategy 1 — Compaction (먼저)**:
- `hooks/scripts/pre-compress.js` 신설 또는 확장
- 대화 길이 임계값(예: 50 turn 또는 토큰 80% 점유) 도달 시 자동 trigger
- 압축 방식: gemini-3-flash-lite로 "지금까지 결정 + 미해결 항목 + 핵심 코드 위치" JSON 추출
- 압축 결과는 `lib/gemini/context-fork.js`의 snapshot으로 저장 (`docs/.pdca-snapshots/`)
- 사용자 명시 disable: `BKIT_COMPACTION=false`

**Strategy 2 — Tool-result clearing**:
- 대용량 Read 결과는 raw 보관 대신 "요약 + 위치(file:line)" 만 보존
- `hooks/scripts/after-tool.js` 확장: tool=Read이고 result_size > N KB 시 자동 요약
- 결정/규칙은 보존, 일회성 raw는 휘발

**Strategy 3 — Memory tool 통합** (S5에서 가장 큰 작업):
- `lib/core/agent-memory.js` ↔ Gemini Auto Memory inbox(PR #26338의 `extraction.patch` 계약) 매핑 실제 구현
- `lib/core/agent-memory.js` 신규 메서드: `exportAsExtractionPatch()`, `importFromExtractionPatch(patch)`
- ENH-3 mapping 문서 §6 "통합 시나리오 5건" 중 최우선 1건 구현 (사용자 결재로 선택)
- 21 agent 기존 memory file 무손상 보장 + Auto Memory inbox와 양방향 sync

**Strategy 4 — Sub-agent isolation** (S3 의존):
- S3 Track A 자연어 dispatch에서 spawn된 agent는 main context를 보지 않고 격리된 context fork로 실행
- 결과만 main으로 반환 (raw context 오염 0)
- `lib/gemini/context-fork.js`의 `forkContext()` 기존 함수 활용

### 6.3 Acceptance Criteria

- [ ] **측정**: 50 turn 대화 시 토큰 사용량 baseline 대비 30% 이상 감소 (실측)
- [ ] Compaction trigger 작동 + `BKIT_COMPACTION=false` 토글 정상
- [ ] Tool-result clearing 후 Read 결과 reference 무결성 (file:line 보존)
- [ ] Memory tool 통합: agent-memory ↔ extraction.patch 1 시나리오 실제 동작

- [ ] Sub-agent isolation: spawn된 agent context dump가 main context로 leak 0건 검증

### 6.4 위험

- **위험 1**: Compaction이 중요한 사용자 컨텍스트를 손실
  - **완화**: 압축 결과를 `.pdca-snapshots/`에 무손실 보존 + 사용자 `/pdca rollback` 명령 추가

  - **완화**: capability flag 기반 게이트 + 스키마 버전 mismatch 시 graceful degradation

  - **완화**: 의존성으로 S3가 선행. S3 Track A 정상 작동 후 S5 진입.
