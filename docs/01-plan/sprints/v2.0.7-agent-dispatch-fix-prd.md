## §1. Beachhead — "이 sprint 가 sells 하는 한 가지"

> **"`gemini -p "Use the code-analyzer agent to ..."` 를 호출하면 실제 code-analyzer 가 응답한다."**

광고-실현 gap 100% → 0%. v2.0.7 사용자가 광고된 21개 specialized agent 모두 의도된 페르소나/도구 권한으로 호출 가능. 기존 generalist fallback 동작은 보존되지만 (기본값 변경 없음) 자연어 dispatch 는 specialist 로 라우팅된다.

### 1.1 Why this beachhead

- 가장 좁은 가치 = 가장 명확한 검증 = 가장 빠른 사용자 신뢰 회복
- 부수 기능 (sub-agent isolation 본격 구현, context engineering 통합) 은 Sprint S5 분리 — scope 보호
- 단일 metric: "21/21 dispatch PASS"

## §2. Job Stories (8건)

### 2.1 JS-1 (en, developer)
**When** I run `gemini -p "Use the code-analyzer agent to review src/utils/parser.js"`,
**I want to** receive a response from the actual code-analyzer agent (gemini-3-pro, read-only tools, temperature 0.3),
**So that** I get the code quality analysis matching the agent's documented persona, not a generic generalist response.

### 2.2 JS-2 (ko, PM)
**언제** `gemini -p "qa-strategist 에이전트로 OAuth 플로우 테스트 계획 만들어줘"` 를 실행할 때,
**나는** qa-strategist 페르소나로 L1~L5 테스트 매트릭스 형식의 응답을 받기를 원한다,
**그래서** PM Agent Team 워크플로우의 specialist 가치를 실제로 활용할 수 있다.

### 2.3 JS-3 (ja, CTO)
**`gemini -p "cto-lead エージェントでv2.1.0アーキテクチャレビューをお願いします"` を実行する時、**
**cto-lead エージェントの実際の応答** (高度な戦略적 reasoning, gemini-3-pro) **を受け取りたい、**
**それにより本物の意思決定支援を得る。**

### 2.4 JS-4 (zh, security architect)
**当我执行** `gemini -p "请用 security-architect agent 进行 OWASP audit"`,
**我希望** 实际的 security-architect agent (read-only tools + OWASP 知识) **响应,**
**这样** 我获得真正的安全审计而不是泛化的安全建议.

### 2.5 JS-5 (en, false-positive 방어)
**When** my prompt contains the word "code-analyzer" as part of a documentation explanation (e.g., "explain what the code-analyzer agent does"),
**I want** the system to ask me whether I meant to dispatch the agent,
**So that** false-positive dispatches do not interrupt my non-dispatch intent.

### 2.6 JS-6 (es, agent 토글)
**Cuando** establezco `BKIT_NATIVE_AGENT_DISPATCH=off` antes de invocar gemini,
**Espero** que el comportamiento sea exactamente el actual (fallback generalist),
**Para** mantener compatibilidad con flujos de trabajo existentes durante la transición.

### 2.7 JS-7 (de, isolation 보장)
**Wenn** ein dispatched Agent (z.B. gap-detector) sein Ergebnis zurückgibt,
**Erwarte ich**, dass das Ergebnis explizit als `[Agent: gap-detector]` Block kennzeichnet ist und meinen Haupt-Conversation-Context nicht verschmutzt,
**Damit** ich Agent-Output von meinem laufenden Dialog unterscheiden kann.

### 2.8 JS-8 (fr, Gemini CLI 업데이트 호환)
**Quand** Gemini CLI v0.43.0+ corrige le bug `LocalSubagentInvocation` en amont,
**Je veux que** bkit détecte automatiquement la capability `hasNativeAgentDispatch` et désactive le bypass,
**Pour** éviter le double-dispatch et bénéficier de la solution upstream.

## §4. User Journey (Before / After)

### 4.1 Before (v2.0.7-upgrade 종료 시점)

```
사용자 입력
$ gemini -p "Use the code-analyzer agent to review src/utils/parser.js"

Gemini CLI 처리
1. LocalSubagentInvocation.execute('code-analyzer', task)
2. agent registry 조회 → 404 (catalog 측은 있지만 invocation 측 broken)
3. generalist agent fallback
4. generalist 가 "코드 리뷰" 일반 응답 생성

사용자 응답

(general advice, code-analyzer 의 read-only tool 정확도 / 보안 패턴 / 아키텍처 분석 없음)

bkit Feature Usage 로그
"Not Used: code-analyzer (Requested but entity not found)"
```

**문제**: 응답이 자연스러워 사용자가 누락 인지 어려움. silent quality degradation.

### 4.2 After (Sprint S3 완료 시점, Track A on)

```

2. agent_name='code-analyzer', task='review src/utils/parser.js' 추출

5. agent .md 페르소나 + task → gemini call → 응답
6. 결과를 [Agent: code-analyzer] 라벨로 감싸서 main flow 로 반환

사용자 응답
```

```
3. Gemini CLI 본체로 위임
```

## §5. Functional Requirements (FR — 13건)

### 5.1 자연어 dispatch matcher

**FR-1** `lib/gemini/agent-dispatch.js` 가 정규식 패턴으로 자연어 dispatch 의도를 매칭한다.
**FR-2** 매칭 시 agent_name (21개 화이트리스트) + task (인용된 문장 또는 다음 문장 ~512 chars) 를 추출한다.
**FR-3** 매처는 8 언어 패턴을 지원한다 (`lib/gemini/agent-dispatch.lang.js`):
  - en: `Use|Invoke|Spawn|Call (the )?<agent>(-agent)? (to|for) <task>`
  - ko: `<agent> (에이전트|서브에이전트)(으?로|에게) <task> (해줘|해주세요|부탁|실행)`
  - ja: `<agent> エージェント(で|に)? <task> (お願い|実行|して)`
  - zh: `(请|让)?用 <agent> agent <task>` / `(请)?调用 <agent> agent <task>`
  - es: `(Usa|Invoca|Llama) (al )?(agente )?<agent> (para|que) <task>`
  - fr: `(Utilise|Appelle|Lance) (l')?(agent )?<agent> (pour|afin de) <task>`
  - de: `(Verwende|Rufe|Starte) (den )?(Agent )?<agent> (um|zu) <task>`
  - it: `(Usa|Invoca|Chiama) (l')?(agente )?<agent> (per) <task>`

**FR-4** agent_name 매칭은 21개 화이트리스트 ↔ exact match only (substring/fuzzy 거부 — false-positive 방지).

### 5.2 MCP spawn_agent 위임

**FR-5** 매칭 성공 시 `mcp/bkit-server.js` 의 `spawn_agent` MCP tool 을 호출한다.
**FR-6** spawn_agent 인자: `{ agent_name, task, context: {dispatch_source: 'bkit-hook', user_prompt_hash}, timeout: agent.timeout_mins * 60 * 1000 }`.
**FR-7** spawn_agent 결과를 `[Agent: <name>]\n<result>\n[End Agent Output]` 형태로 감싸 반환.

### 5.3 토글 및 capability flag

**FR-8** `BKIT_NATIVE_AGENT_DISPATCH` env 4 모드:
  - `auto` (기본): capability flag `hasNativeAgentDispatch` true 시 우회 비활성화, false 시 우회 활성화
  - `on`: 무조건 Track A 우회 사용
  - `off`: 무조건 Gemini CLI 본체로 위임
  - `ask`: 매칭 시 AskUserQuestion 확인 후 진행 (false-positive 방어)

**FR-9** `lib/gemini/version.js` 에 `hasNativeAgentDispatch: isVersionAtLeast('0.44.0')` capability flag 추가 (가정: v0.43.0 = PR 머지 직후, v0.44.0 stable = capability 보장).

### 5.4 False-positive 방어

**FR-10** 사용자가 본 sprint 처음 사용 시 first 2 dispatch 는 AskUserQuestion 으로 확인 → 응답을 `.bkit/agent-memory/dispatch-confirmed.json` 에 기록 → 이후 자동 처리.
**FR-11** 매처가 의심스러운 매칭 (예: agent_name 이 코드 블록 안에 있는 경우, prompt 가 "explain", "what is", "how does" 등 메타 패턴 포함) 시 매칭 거부.

### 5.5 Isolation 보장 (M8 dataFlowIntegrity)

**FR-12** Agent 결과는 `[Agent: <name>]` 라벨 + `[End Agent Output]` 경계로 감싸진다 — main conversation context 와 명시적 boundary.
**FR-13** spawn_agent 의 결과 중 agent 가 내부적으로 사용한 system prompt / agent .md frontmatter 는 main context 에 노출되지 않는다 (raw context leak 방지).

## §6. Non-Functional Requirements (NFR)

| ID | NFR | 목표 |
|---|---|---|
| NFR-1 | Latency overhead (matcher only) | < 10ms per prompt (regex matching) |
| NFR-2 | Latency overhead (dispatch full) | < 100ms additional vs Gemini 본체 dispatch (matcher + MCP call setup) |
| NFR-3 | Memory footprint | < 1MB per dispatch (agent .md 캐시 포함) |
| NFR-4 | Test coverage (agent-dispatch.js) | line ≥ 80%, branch ≥ 70% |
| NFR-5 | Test coverage (agent-dispatch.lang.js) | line ≥ 90% (정규식이라 simpler) |
| NFR-6 | Failure mode | matcher exception 시 silent fallback (Gemini CLI 본체로 위임) — 사용자 차단 0 |
| NFR-7 | Logging | dispatch 발생/실패 모두 `.bkit/audit/dispatch-{date}.log` 에 기록 |
| NFR-8 | Multilingual coverage | 8 언어 패턴 각 ≥ 1 test case |
| NFR-9 | Cost | dispatch matcher 자체 cost 0 (regex). agent spawn 은 기존과 동등 (model 비용은 agent .md frontmatter 의 model 에 따름) |
