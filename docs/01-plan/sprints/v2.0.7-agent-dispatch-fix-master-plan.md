## §0. Executive Summary

### 0.1 Mission

bkit이 광고하는 21개 specialized agent (`code-analyzer`, `security-architect`, `gap-detector`, `pdca-iterator`, `qa-strategist`, …) 가 **자연어 dispatch (`gemini -p "Use the X agent to ..."`)** 호출에서 실제로 의도된 페르소나/도구 권한으로 응답하도록 한다. 현재 동일 호출은 Gemini CLI 본체 `LocalSubagentInvocation.execute` 의 registry 조회 실패로 `generalist` agent fallback 처리되어 광고와 실제 동작 사이의 silent quality degradation 이 발생한다.

### 0.2 Anti-Mission (scope 외)

- 21개 agent 인벤토리/페르소나/도구 권한 변경 — `agents/*.md` frontmatter 보존
- Context Engineering 4-strategy 통합 — Sprint S5 (`v2.0.7-context-engineering-integration`) 책임
- Sub-agent isolation 본격 구현 — Sprint S5 책임 (본 sprint는 Track A 우회를 통해 main context 오염 0만 보장)
- L2 baseline 9건 보안 fail / 74건 일반 baseline — Sprint S2 / S6 책임
- Gemini CLI fork repo 자체 운영 — 본 sprint는 upstream PR 1건 제출까지

## §1. Context Anchor

### 1.1 WHY (왜 진행하는가)

**1차 RCA 결과 (v2.0.7-upgrade Sprint W4, 2026-05-14)**:

```
[v0.39.1 + v0.42.0 동일 결과]
$ gemini -p "Use the code-analyzer agent to evaluate: var x = 1; eval(input);"
→ "code-analyzer agent was not found"
→ generalist fallback
→ stack: at async LocalSubagentInvocation.execute
   file:///.../@google/gemini-cli/bundle/chunk-7VVHSNDQ.js:309875:22
   code: 404
→ bkit Feature Usage: "Not Used: code-analyzer (Requested but entity not found)"
```

**중요 사실**:
- `gemini agents list` → bkit 21 agent + 내장 3개 = 24 정상 표시. **catalog 정상**.
- `agents/*.md` 21 file 모두 frontmatter 완전 (name/model/tools/temperature/max_turns).
- `mcp/bkit-server.js:321` → `spawn_agent` MCP tool이 이미 `enum: Object.keys(AGENTS)` 로 21개 등록.
- 즉 **catalog 측은 정상, invocation 측만 broken**. Gemini CLI 본체 `LocalSubagentInvocation` 의 두 코드패스(catalog vs invocation) 중 invocation 측 registry 조회만 fail.

**왜 v2.0.7 안에서 해결**:
- 21 agent 가치 = 0% 상태로 v2.1.0까지 1 minor cycle 더 지속 시 사용자 신뢰 침식
- Sprint S5 (Context Engineering Sub-agent isolation) 가 본 sprint의 dispatch 위에서 동작 — 별도 minor 분리 시 통합 비용 2배
