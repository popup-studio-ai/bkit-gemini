## §0. Executive Summary

### 0.1 Mission

bkit Gemini Extension의 자동화 한도를 **L2 → L4 (Autonomous)**까지 끌어올려, v2.0.7 Debt Resolution Program의 후속 6 sprint(S1~S6)가 사용자 명시 정책 `/sprint start + /control level L4 완전 자동 모드`로 진행될 수 있도록 한다. 동시에 Gemini CLI 본체의 hook contract를 항구적으로 개선하기 위한 PR을 병행 제출하여 임시 워크어라운드를 거부한다.

### 0.2 Anti-Mission (해서는 안 되는 일)

1. **L4 만능키 + 무절제 destructive 허용**: 임시방편으로 모든 tool을 무조건 'allow'로 만들고 사용자 결재를 우회하는 패턴 거부. destructive ops는 trust score / level과 무관하게 영구 deny.
2. **Track A 단독 완료**: hook 우회만으로 L4를 "동작하는 것처럼" 보이게 하고 Gemini CLI 본체 contract 결함을 방치하는 패턴 거부. 반드시 Track B PR을 짝으로 제출한다.
3. **bootstrap paradox 우회**: S7 sprint 자신을 L4로 실행하려는 시도 거부. S7은 메타-부트스트랩이므로 L2 + 사용자 phase 결재로만 진행 (자기 자신을 활성화하는 sprint은 자기 자신을 사용할 수 없다).
4. **무회귀 위반**: 기존 L2 사용자 (전 bkit 사용자)의 동작 변경 0건이 strict 조건. L4 코드 경로는 명시적 opt-in 시에만 활성화.
5. **audit log 누락**: L4 모든 자동 결정은 영구 audit 기록 의무. 없으면 사용자 신뢰 침식 + 회귀 추적 불가.

## §5. Sprint Split Recommendation

S7은 단일 sprint로 진행한다. 분할 검토 결과:

### 6.2 Pre-mortem (만약 sprint가 fail 한다면…)

가상 시나리오 3개:

1. **"L4 모드 진입 후 첫 sprint에서 production code 의도치 않게 삭제"** — 원인: destructive 패턴 leak. 예방: AC-T3 10 시나리오 + AC-T5 checkpoint + R7-E AST 검증. Recovery: `/control rollback` 무결재.
2. **"trust_score가 boundary에서 oscillation해서 사용자가 L4↔L2 깜빡임 경험"** — 원인: cooldown 없음. 예방: R7-C-2 24h cooldown 강제. Detection: `automationHistory` 분석 (24h 내 2회 이상 level 변경 시 alert).
3. **"Gemini CLI 본체가 'allow' 결정을 무시해서 Track A 우회가 작동 안 함"** — 원인: hook contract 자체 결함. 예방: F7-1 RCA에서 reproducer로 확정. Mitigation: AC-T1을 위해 alternative path (sandbox-mode 우회 또는 BKIT_LEVEL=L4 환경변수 force-set) 검토 + Track B PR fast-track.
