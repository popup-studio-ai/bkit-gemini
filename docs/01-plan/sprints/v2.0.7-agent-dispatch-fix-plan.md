## §2. Decisions (D1~D12)

### D1. 2-Track 병행 (Track A + Track B) vs 단일 Track

**대안**:
- A1: Track A (bkit hook 우회) 만 — 임시방편 영구화
- A2: Track B (Gemini CLI 본체 PR) 만 — 머지 시점 불확실, 사용자 즉시 차단 해제 불가
- A3: 2-Track 병행 — 즉시 가치 + 항구 해결 양립

**선택**: **A3** (Master Plan §4.2 결정 + 본 sprint 핵심 원칙).
**근거**: 두 track 함께 진행해야 임시방편이 아닌 근본 해결. Track A 단독은 v0.43.0+ 본체 변경과 충돌 위험, Track B 단독은 머지 거부/지연 시 사용자 가치 0.

### D2. bkit hook 진입점 — `before-model` vs `before-agent`

**대안**:
- B1: `hooks/scripts/before-model.js` — model 호출 전 prompt 가공
- B2: `hooks/scripts/before-agent.js` — agent 호출 전 (이미 dispatch 결정된 후)

**선택**: **B1** (`before-model.js`).
**근거**: dispatch 가 model 호출 전 결정되어야 LocalSubagentInvocation 우회 가능. before-agent 는 이미 agent 결정된 후 hook — 너무 늦음.
**파일**: `hooks/scripts/before-model.js` (현 218L) + 신규 `lib/gemini/agent-dispatch.js` 로 로직 분리.

### D3. 매처 모듈 분리 — inline vs 신규 lib 모듈

**대안**:
- C1: `hooks/scripts/before-model.js` 인라인 (단순)
- C2: `lib/gemini/agent-dispatch.js` 신규 모듈 (재사용 + 테스트)
- C3: `lib/gemini/hooks.js` 확장 (기존 hooks 도우미 모듈)

**선택**: **C2**.
**근거**:
- 재사용성: MCP server / CLI 양쪽에서 호출 가능
- 테스트 가능성: hook 본체와 매처 로직 분리 → unit test 작성 가능 (NFR-4 coverage ≥ 80%)
- 8 언어 카탈로그 분리 (`agent-dispatch.lang.js`): 점진 보강 가능

**파일**: `lib/gemini/agent-dispatch.js` (~300L) + `lib/gemini/agent-dispatch.lang.js` (~150L).

### D4. 정규식 vs LLM-based intent classification

**대안**:
- D1: 정규식 매처 (deterministic, low cost)
- D2: gemini-3-flash-lite intent classifier (1 LLM call/prompt)
- D3: 하이브리드 (regex 우선 + ambiguous case 만 LLM)

**선택**: **D1** (정규식).
**근거**:
- NFR-1 < 10ms latency — LLM call 은 수백 ms
- NFR-9 cost 0 — LLM 추가 비용 없음
- false-positive 는 FR-10/11 (확인 + 메타 패턴) 으로 방어
- 미래에 D3 하이브리드 전환 가능 (`agent-dispatch.js` 의 인터페이스만 확장)

### D5. agent_name 매칭 정확도 — exact vs fuzzy

**대안**:
- E1: exact match (21개 화이트리스트)
- E2: fuzzy match (Levenshtein distance, 예: "code-analyser" → "code-analyzer")
- E3: prefix match (예: "code-an" → "code-analyzer")

**선택**: **E1** (exact match).
**근거**:
- false-positive 최소화 — 사용자가 "코드 분석 (code analysis)" 본문에 쓸 때 매칭 안 됨
- 21개는 작은 set — typo 는 학습으로 해결 (FR-10 메모리)
- AC-A4 (21개 외 매칭 거부) 직접 충족

### D6. 토글 모드 4종 vs 2종

**대안**:
- F1: on/off 2종
- F2: auto/on/off 3종
- F3: auto/on/off/ask 4종

**선택**: **F3** (4종).
**근거**:
- `auto`: 일반 사용자 (기본)
- `on`: 강제 우회 (Track B 머지 후 test 또는 사용자 명시 의도)
- `off`: 강제 비활성화 (Track B 머지 후 기본값으로 전환 또는 사용자 opt-out)
- `ask`: false-positive 우려 시 확인 모드 (debugging / 신뢰성 우선)
- FR-8 정의

### D7. boundary 라벨 형식

**대안**:
- G1: `[Agent: <name>]\n...\n[End Agent Output]` (text)
- G2: JSON `{"agent": "<name>", "output": "..."}` (machine-readable)
- G3: Markdown blockquote `> **Agent: <name>**\n> ...`

**선택**: **G1**.
**근거**:
- 사용자 가독성 최우선 (M10)
- LLM context 에서 boundary 인식 명확 (`[End Agent Output]` sentinel)
- JSON 은 main response 가 user-facing 일 때 noise
- 향후 G2 JSON 모드는 `--format=json` flag 로 확장 가능 (scope 외)

### D8. spawn_agent timeout 처리

**대안**:
- H1: 항상 5min (MCP default)
- H2: agent .md frontmatter `timeout_mins` 사용
- H3: 사용자 prompt 의 명시 (예: "with 10min timeout") + frontmatter fallback

**선택**: **H2**.
**근거**:
- agent .md 가 의도된 timeout — pdca-iterator 등 long-running agent 는 더 긴 timeout
- 사용자 prompt parsing 복잡도 회피 (scope 외)
- FR-6 명시

### D9. 메타 패턴 거부 (FR-11)

**대안**:
- I1: 정규식 negative lookahead (`(?!explain|what is|how does)`)
- I2: 후처리 (매칭 후 prompt context 검사)
- I3: 메타 패턴 + 백틱 안 위치 검사

**선택**: **I2 + I3 조합**.
**근거**:
- I1 의 부정 lookahead 는 8 언어 × 다양한 메타 패턴 → 정규식 복잡도 폭증
- I2 후처리는 매칭 결과 검사 — agent_name 이 백틱 코드 블록 안인지, 메타 단어 (`설명`, `what is`, ...) 가 prompt 앞 부분에 있는지 검사
- 메타 단어 카탈로그는 `agent-dispatch.lang.js` 의 `META_PATTERNS` 로 분리

### D10. agent-memory 저장 위치

**대안**:
- J1: `.bkit/agent-memory/dispatch-confirmed.json` (project scope)
- J2: `~/.gemini/agent-memory/bkit/dispatch-confirmed.json` (user scope, global)
- J3: 둘 다

**선택**: **J1** (project scope).
**근거**:
- 프로젝트마다 dispatch 패턴 다를 수 있음
- 사용자 신뢰는 프로젝트별 학습이 적절
- 미래에 J3 통합 (Sprint S5 의 Memory tool 통합 시) 으로 확장 가능

### D11. Track B reproducer 위치

**대안**:
- K1: `tests/fixtures/agent-dispatch-repro/` (테스트 픽스처)
- K2: Gemini CLI fork repo 의 `test/repro/` (외부 분리)
- K3: Gist 또는 standalone repo

**선택**: **K1 + K2 dual**.
**근거**:
- K1: bkit 자체 테스트에 활용 (RCA 검증 + 향후 회귀)
- K2: PR 본문에 인용 가능한 minimal repro
- K3: 의 추가 maintenance 회피 — fork repo 의 test/ 폴더로 충분

### D12. capability flag 버전 기준

**대안**:
- L1: `hasNativeAgentDispatch: isVersionAtLeast('0.43.0')` (PR 머지 직후 가정)
- L2: `hasNativeAgentDispatch: isVersionAtLeast('0.44.0')` (다음 stable 보장)
- L3: 동적 detection (실제 호출 시 fail/success 학습)

**선택**: **L2** (0.44.0).
**근거**:
- 0.43.0 은 PR 머지 시점 — minor patch 가 있을 수 있음 (regression 가능성)
- 0.44.0 stable 에서 capability 보장 시점이 안전
- L3 동적 detection 은 첫 호출 fail 시 1회 generalist fallback → 사용자 차단 1회 발생 (FR-6 silent fallback 원칙 위배)
- FR-9 명시

### 3.2 양 Track 의 관계

- **Track A**: 즉시 사용자 가치 보존 (today). 우회 모듈은 capability flag 로 자동 무효화 가능.

- **통합**: regression suite 가 두 track 모두 검증. capability flag 로 자동 전환.

### 3.3 외부 의존성 격리

- Gemini CLI PR 의 머지 거부/지연 시에도 sprint 완료 가능 (Track A 단독으로 사용자 가치 100%)
