## §1. Design Overview

### 1.1 설계 목표

Gemini CLI Extension 환경에서 **L4 Autonomous 모드**를 안전하게 동작시킨다. 이 sprint가 다른 모든 sprint(S1~S6)의 prerequisite이며, 잘못 설계 시 destructive op가 자동 승인되어 사용자 데이터 손상 위험이 발생한다. 따라서 모든 설계 결정은 **안전 우선**(safety-first)을 따르고, 각 결정의 안전 가드를 Design에 명시한다.

## §2. Module Design — `lib/core/trust-score.js` (C1)

### 2.1 책임
- trust_score 동적 조정
- automation level 전환 가능 여부 판정
- auto-downgrade 트리거 감지
- recentDecisions sliding window 유지

### 2.2 인터페이스

```js
const TrustScoreManager = require('lib/core/trust-score');
const mgr = new TrustScoreManager(projectDir);

mgr.getScore();                        // → number [0, 100]
mgr.getLevel();                        // → number [0, 4]
mgr.canEscalate(targetLevel);          // → boolean
mgr.recordDecision({                   // 부수효과: persist
  type: 'allow' | 'deny' | 'rejection',
  tool: 'Read',
  timestamp: Date.now(),
  durationMs: 12,
  rejected: false
});
mgr.shouldDowngrade();                 // → null | { from, to, reason }
mgr.escalate(targetLevel, reason);     // → boolean (성공 여부)
mgr.downgrade(targetLevel, reason);    // 부수효과: persist
mgr.emergencyStop();                   // 즉시 L0 + emergencyStop=true
mgr.releaseEmergencyStop();            // sessionEnd hook에서 정리
```

### 2.3 알고리즘 (Plan §6.1 기반)

**Score 조정**:
- success allow: score = min(100, score + 5)
- explicit rejection: score = max(0, score - 10)
- (allow but later rejected): score = max(0, score - 5) (절반 페널티)

**Escalation 조건**:
| Target Level | trustScore 조건 | Recent 조건 | Cooldown |
|-------------|----------------|------------|---------|
| L1 | >= 20 | — | no |
| L2 | >= 40 | recent 5 rejections <= 2 | no |
| L3 | >= 60 | recent 5 rejections <= 1 | no |
| L4 | >= 80 | recent 10 rejections == 0 | no active cooldown |

**Auto-downgrade**:
- 5 consecutive rejections → L2 (24h cooldown)
- Hard deny (destructive 시도) → L3 (no cooldown)
- emergency_stop → L0 (session 종료까지)

### 2.4 Persistence

`recordDecision` / `escalate` / `downgrade`마다 `.bkit/runtime/control-state.json` 자동 저장 (debounce 100ms).

### 2.5 Unit Test 매트릭스 (AC-T13)

| Test | 시나리오 | 기대 |
|------|---------|-----|
| TS-01 | 신규 init | score=40, level=2 |
| TS-02 | 5 success allow | score=65 |
| TS-03 | 5 success → 10 rejection | score=15 |
| TS-04 | 100 success cap | score=100 |
| TS-05 | 0 score 0 cap | score=0 |
| TS-06 | canEscalate(4) @ score=80, recent 10 rejection=0 | true |
| TS-07 | canEscalate(4) @ score=85, recent 10 rejection=1 | false |
| TS-08 | 5 연속 rejection → downgrade L2 | level=2, cooldown set |
| TS-09 | 100ms boundary boundary | second decision ignored |
| TS-10 | emergencyStop() → L0 | level=0, emergencyStop=true |
| TS-11 | Hard deny → downgrade L3 | level=3, no cooldown |
| TS-12 | persist + reload roundtrip | state preserved |

**Mutation testing (H-5, AC-T13)**:
- Tool: `stryker-js` (`@stryker-mutator/core` + `@stryker-mutator/jest-runner`)
- 적용 범위: `lib/core/trust-score.js`, `lib/core/cmd-parser.js`, `lib/core/checkpoint.js`

## §3. Module Design — `lib/core/control-state.js` (C2)

### 3.1 책임
- control-state.json read/write 캡슐화
- v1.0 → v1.1 자동 마이그레이션
- atomic write (temp file + rename)

### 3.2 인터페이스

```js
const cs = new ControlStateManager(projectDir);
cs.load();          // 자동 마이그레이션 (v1.0 → v1.1)
cs.get(field);
cs.set(field, value);
cs.save();          // atomic write
```

### 3.3 v1.0 → v1.1 마이그레이션

```js
function migrate_v10_to_v11(state) {
  if (state.version === '1.1') return state;
  return {
    ...state,
    version: '1.1',
    automationHistory: state.automationHistory || [],
    recentDecisions: state.recentDecisions || []
  };
}
```

기존 필드는 모두 보존. AC-T6 (L2 무회귀) 보장.

### 3.4 Atomic Write (M-2)

```js
function atomicSave(filePath, content) {
  const tmpPath = filePath + '.tmp.' + process.pid + '.' + Date.now();
  fs.writeFileSync(tmpPath, content);
  fs.renameSync(tmpPath, filePath);  // atomic on POSIX (macOS/Linux), best-effort on Windows
}
```

**Cross-OS race condition 처리 (M-2)**:
- POSIX (macOS/Linux/WSL): `rename(2)` atomic 보장
- Windows: NTFS는 atomic rename 미보장 → file locking 추가 (proper-lockfile npm 또는 자체 mutex)
- bkit-gemini는 **단일 hook process 가정** (Gemini CLI는 hook을 sequential 실행) → race condition 발생 가능성 매우 낮음. 단 안전을 위해 `proper-lockfile` 또는 sentinel-file 패턴 적용.

## §3.5 Module Design — `lib/core/checkpoint.js` (C5, C-3 신규)

### 3.5.1 책임 (AC-T5 매핑)
- 모든 코드 수정 전 자동 checkpoint 생성
- 사용자 결재 없이 rollback 가능 (`/control rollback`)
- 90d 자동 retention

### 3.5.2 인터페이스

```js
const cm = new CheckpointManager(projectDir);
cm.create(operation, affectedFiles);  // → checkpointId (uuid)
// 부수효과: .bkit/checkpoints/{timestamp}-{op}.json 저장
//   { id, timestamp, operation, files: [{ path, beforeContent, beforeMTime }], reason }

cm.list({ from, to, limit });          // → [{ id, timestamp, operation, files[] }]
cm.get(checkpointId);                  // → checkpoint detail
cm.rollback(checkpointId);             // → boolean (성공 여부)
//   - 각 file의 beforeContent를 restore
//   - 현재 mtime이 checkpoint.beforeMTime와 다르면 conflict → 사용자 확인 (단, L4에서는 자동 conflict 해결 X — quarantine)
cm.purge(olderThanDays);               // 90d retention
```

### 3.5.3 Storage 스키마 (`.bkit/checkpoints/{checkpointId}.json`)

  "id": "uuid-v4",

    {
      "path": "/abs/path/to/file.js",

### 3.5.4 Rollback 안전 가드 (AC-T5)

| 시나리오 | 처리 |
|---------|-----|
| 정상 rollback | beforeContent 그대로 write |

| 디스크 공간 부족 | rollback 실패 + 명시적 에러 |

## §4. Module Design — `lib/core/cmd-parser.js` (C3, D7)

### 4.1 책임
- Shell 명령을 AST로 파싱 (heredoc, escape, variable expansion 인식)
- Destructive 패턴 검출
- 우회 시도 차단 (base64 디코드, eval, $(), backtick, ...)

### 4.2 인터페이스

```js
const parser = new CmdParser();
parser.parse('rm -rf /tmp/foo');
// → { commands: [{ executable: 'rm', args: ['-rf', '/tmp/foo'], destructive: true, pattern: 'rm-recursive' }] }

parser.parse('cat << EOF | bash\nrm -rf /\nEOF');
// → { commands: [{ executable: 'bash', stdin: 'rm -rf /\n', destructive: true, pattern: 'heredoc-rm-recursive' }] }

parser.isDestructive(cmd);    // → boolean
parser.detectBypass(cmd);     // → null | 'heredoc' | 'base64' | 'eval' | 'subshell'
```

### 4.3 Destructive 패턴 매트릭스 (H-2 credential 패턴 포함 16종)

| 패턴 ID | Regex (AST 노드 기반) | 예시 |
|---------|---------------------|------|
| D-RM-RF | `rm` + arg `-rf` or `-r` + `-f` | `rm -rf /tmp` |
| D-FORCE-PUSH | `git push` + arg `--force` or `-f` | `git push -f origin main` |
| D-DROP-TABLE | SQL `DROP TABLE` | `drop table users;` |
| D-TRUNCATE | SQL `TRUNCATE` | `truncate users;` |
| D-DELETE-NO-WHERE | SQL `DELETE FROM` 없이 `WHERE` | `delete from users;` |
| D-SUDO | `sudo` (any) | `sudo apt install ...` |
| D-CHMOD-777 | `chmod` + 777 | `chmod 777 ~/.ssh` |
| D-ETC-WRITE | path 시작 `/etc/` + write op | `echo x > /etc/hosts` |
| D-SSH-WRITE | path 시작 `~/.ssh/` + write op | `> ~/.ssh/config` |
| D-GPG-DISABLE | `gpg --delete-key` 등 | `gpg --batch --yes --delete-key X` |
| **D-ENV-WRITE** | path matches `.env*` + write op (H-2) | `echo SECRET=x > .env` |
| **D-PRIVATE-KEY-WRITE** | path matches `id_rsa*`, `*.pem`, `*.p12`, `*.kdbx` + write/delete | `rm id_rsa.pub` |
| **D-NPM-FORCE-PUBLISH** | `npm publish` + `--force` (H-2) | `npm publish --force` |
| **D-GIT-CONFIG-UNSET** | `git config --unset` + credential.* (H-2) | `git config --unset credential.helper` |
| **D-RM-DOT-GIT** | `rm` + `.git` 디렉터리 | `rm -rf .git` |
| **D-DROP-DATABASE** | SQL `DROP DATABASE` | `drop database production;` |

**Cross-OS 주의**: Windows PowerShell 환경에서는 `Remove-Item -Recurse -Force`, `Set-Acl`, `icacls`도 destructive로 검출 필요 (cmd-parser가 shell 자동 감지).

### 4.4 Bypass 검출 (H-1, 11종)

| Bypass | 검출 방법 |
|--------|---------|
| heredoc (`<<EOF`) | AST에서 stdin 노드 추출 후 재귀 파싱 |
| base64 | `echo <base64> \| base64 -d \| bash` 패턴 매치 후 디코드 검사 |
| eval | `eval` keyword 자체를 destructive로 간주 |
| subshell (`$()`, backtick) | 중첩 명령 재귀 파싱 |
| variable expansion | `$VAR` resolve 시도 (best-effort), unresolved는 conservative deny |
| **exec** (H-1) | `exec` keyword 검출 → 자식 명령 재귀 파싱 |
| **xargs** (H-1) | `xargs` 검출 → -I/-n 인자로 들어가는 명령 분석 |
| **fd redirection** (H-1) | `exec 3<file` + `<&3 \| bash` 패턴 검출, stdin 재귀 분석 |
| **IFS 변조** (H-1) | `IFS=` 또는 `IFS=$'...'` 명시 변경 시 word splitting 재해석 |
| **ANSI/control char** (H-1) | `\x08`, `\b`, `\r`, ANSI escape 시퀀스 제거 후 재검출 (`r\x08m -rf` → `rm -rf`) |
| **alias/function override** (H-1) | `alias` / `function` 정의 + 이후 호출 추적 (`function rm() { /bin/rm -rf "$@"; }`) |

**검출 전략 (D16' revised, Wave 0 Day 1)**:
1. **자체 mini-parser only** (regex + token + escape unwind, 의존성 0): 위 11종 bypass + 16 destructive 패턴 모두 검출 (<5ms)
2. **bash-parser npm 의존성 취소**: maintenance 9년 전 abandoned (Wave 0 RCA §4) → bkit이 직접 maintain
3. **의심 명령 → conservative deny**: 자체 parser가 ambiguous 판정한 경우 deny + 사용자 결재 요청 (L3+ 모드, L4 모드는 deny 후 quarantine)
4. **v2.1.0+ 재평가**: 만약 자체 parser CP-01~31에서 한계 발견 시 v2.1.0에서 `@ericcornelissen/bash-parser` 활성 fork 도입 검토

**Cross-OS 보장**: pure JavaScript regex + token 분석 → macOS / Windows (PowerShell + WSL + Git Bash) / Linux 동일 동작. 외부 의존성 0.

## §5. Hook Design

### 5.1 `hooks/scripts/before-tool.js` (C6 강화)

기존 구조 보존, `evaluateAutomation()` 함수만 추가:

```js
function evaluateAutomation(toolCall, level, state) {
  // L0: 모든 호출 deny (Manual)
  if (level === 0) return { decision: 'ask', reason: 'L0 manual mode' };

  // L1~L2: 기존 동작 (whitelist + sensitive prompt)
  if (level <= 2) return evaluateLevel12(toolCall);

  // L3~L4: 자동 'allow' 채널 (이 sprint 신설)
  const parser = new CmdParser();
  const cmdAnalysis = parser.parse(toolCall.command || '');

  // Hard deny check (L4 무관, D6)
  if (parser.isDestructive(cmdAnalysis)) {
    state.mgr.recordDecision({ type: 'rejection', tool: toolCall.tool, ... });
    state.mgr.downgrade(3, 'destructive op detected (D6)');
    return { decision: 'deny', reason: `L4-hardguard: ${cmdAnalysis.pattern}` };
  }

  // Whitelist check (D5)
  const whitelist = ['Read', 'Glob', 'Grep', 'LSP', 'list_directory', 'read_file', ...];
  if (whitelist.includes(toolCall.tool)) {
    state.mgr.recordDecision({ type: 'allow', tool: toolCall.tool, ... });
    return { decision: 'allow', reason: 'L3+ whitelist' };
  }

  // L4 + trust_score 충족: 자동 'allow'
  if (level === 4 && state.mgr.getScore() >= 80) {
    state.mgr.recordDecision({ type: 'allow', tool: toolCall.tool, ... });
    return { decision: 'allow', reason: 'L4 auto-approve' };
  }

  // L3: 위 조건 미충족 시 ask
  return { decision: 'ask', reason: 'L3 fallback' };
}
```

```js
async function processBeforePhase(currentPhase, nextPhase, state) {
  const level = state.mgr.getLevel();

  // L0~L2: 사용자 결재 prompt
  if (level <= 2) {
    return {
      decision: 'ask',
      prompt: `Phase 전환: ${currentPhase} → ${nextPhase}. 진행하시겠습니까?`
    };
  }

  // L3~L4: Agent dispatch — sprint-qa-flow agent 가 M1~M10 게이트 평가
  // Hook은 async response 지원 (Gemini CLI hook spec). 평가 시간 1~5초 예상.
  const gateReport = await dispatchSprintQaFlow({
    sprintName: state.currentSprint,
    fromPhase: currentPhase,
    toPhase: nextPhase,
    targetGates: ['M1', 'M2', 'M3', 'M4', 'M5', 'M6', 'M7', 'M8']
  });

  if (gateReport.overall !== 'PASS') {
    AuditLogger.append({
      timestamp: new Date().toISOString(),
      event: 'phase-transition-blocked',
      from: currentPhase, to: nextPhase,
      failed: gateReport.failed,
      decision: 'deny'
    });
    return {
      decision: 'deny',
      reason: `Quality Gate fail: ${gateReport.failed.join(', ')}. sprint-qa-flow agent report: ${gateReport.summary}`
    };
  }

  // L4: 자동 진행
  AuditLogger.append({
    timestamp: new Date().toISOString(),
    event: 'phase-transition',
    from: currentPhase, to: nextPhase,
    decision: 'allow',
    reason: 'L4 auto-advance, agent dispatch gates PASS',
    agentReport: gateReport
  });
  return { decision: 'allow', reason: 'L4 phase auto-advance' };
}

// Helper: MCP spawn_agent으로 sprint-qa-flow agent 호출
async function dispatchSprintQaFlow(context) {
  const mcpClient = require('lib/mcp/client');
  const result = await mcpClient.callTool('spawn_agent', {
    agent_name: 'sprint-qa-flow',
    task: `Evaluate Quality Gates ${context.targetGates.join(',')} for sprint=${context.sprintName} transition ${context.fromPhase}→${context.toPhase}`,
    context,
    timeout: 60000  // 60s — gate 평가 충분
  });
  return JSON.parse(result.content[0].text);
}
```

**D15 결정 근거**:
- bkit `agents/qa-strategist.md` + sprint-qa-flow agent는 이미 M8 8-Layer dataFlowIntegrity 평가에 특화
- Hook 내 sync evaluation은 hook 코드를 비대하게 만들고 agent의 도메인 전문성 손실
- 1~5초 latency는 phase 전환 빈도(sprint당 6회)에 비해 무시 가능
- Agent timeout 60s + fallback (timeout 시 deny + 사용자 알림)

**Cross-OS 주의**: `spawn_agent` MCP 호출은 pure JSON-RPC over stdio → 모든 OS 동일.

## §6. 8-Layer dataFlowIntegrity (M8 게이트)

각 Layer는 dispatch와 동일한 8-Layer를 control-flow에 적용:

| Layer | 단계 | Component | Validation |
|-------|------|-----------|-----------|
| L1 UI | User prompt `gemini -p "..."` | Gemini CLI | input sanitization |
| L2 Client | Hook entrypoint | `before-tool.js` | hook signature check |
| L3 API | evaluateAutomation() | `before-tool.js:evaluateAutomation` | level/state validation |
| L4 Validation | AST + whitelist + trust_score | `cmd-parser.js`, `trust-score.js` | destructive check 100% |
| L5 DB | control-state.json + audit log | `control-state.js`, `audit-log.js` | atomic write |
| L6 Response | decision 반환 | `before-tool.js` | schema validation |
| L7 Client | Tool 실행 결과 | Gemini CLI tool runner | result schema |
| L8 UI | 사용자 화면 표시 | session-start hook + AuditLogger | leak detector |

**s1Score 계산**: 각 layer pass / fail의 8/8 = 100. M8 PASS 조건.

**Leak detector (AC-T10)**:
- before-tool에서 결정 후, after-tool까지 main context로 hook 내부 state(trust_score 알고리즘 상세, AST 트리 등)가 전달되지 않음을 검증
