### 1.2 Sprint vs PDCA 차이

| 항목 | PDCA (per-feature) | **Sprint (본 sprint)** |
|---|---|---|
| 흐름 | 9-Phase (pm/plan/design/do/check/act/qa/report/archive) | **8-Phase (prd/plan/design/do/iterate/qa/report/archive)** |
| 단위 | 단일 feature | **4 features (F1~F4) 컨테이너** |
| PR 단위 | feature당 1 PR | **sprint 전체 단일 PR** |
| 결재 | feature당 결재 | **sprint 전체 D1/D3/D4/D5 통합 결재** |
| Auto-Pause | 없음 | **4 triggers (QUALITY_GATE_FAIL / ITERATION_EXHAUSTED / BUDGET_EXCEEDED / PHASE_TIMEOUT)** |
| Gate | 단순 phase 전이 | **M1~M10 + dataFlowIntegrity 7-Layer** |
| Carry items | feature별 | **sprint별 (다음 sprint 인계)** |

### 1.3 본 sprint의 본질적 가치

> 본 sprint는 PDCA 9-Phase per-feature 사이클을 5회 (v0.40.0/v0.41.1/v0.41.2/v0.42.0-preview/v0.42.0 stable) 별도 처리할 경우 발생하는 **미머지 plan 누적 부담**과 **결재 분산**을 sprint 8-Phase × 1 컨테이너로 압축하여 해소한다. 결과적으로:
>
> - **PR 5건 → 1건** (D1 시나리오 A 활성화)
> - **사용자 결재 ~20건 → 4건** (D1/D3/D4/D5)
> - **사이클 시간 ~25h → ≤ 5h** (5 cycle × ~5h × redundancy → 단일 통합)
> - **token 예산 ~250K → < 50K** (sprint 재사용 압축)
> - **회귀 검증 ~5회 → 1회** (21 agent smoke + baseline 단일 실행)
