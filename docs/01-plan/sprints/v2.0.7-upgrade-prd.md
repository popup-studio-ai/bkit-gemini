## §3. Job Stories (6건, ENH-1~6 기준)

JTBD (Jobs To Be Done) format: "When [situation], I want to [motivation], so I can [expected outcome]."

### 3.1 JS-1 (ENH-1 Conductor 인용)

> **When** I evaluate bkit's PDCA approach as a CTO considering enterprise adoption,
> **I want to** see an external official source validating that bkit's philosophy aligns with Google's own context-driven development direction,
> **so I can** justify the architectural choice to my team without relying solely on bkit's self-narrative.

**현재 상태**: bkit이 self-narrative만 제공. Google Conductor 블로그(2025-12-17)와 동형성 *명시 인용 없음*.

**해결 (ENH-1)**: README.md v2.0.7 Highlights + GEMINI.md 상단에 Conductor 인용 + 4 동형성 매핑 표.

### 3.2 JS-2 (ENH-2 M6 정당화 명문화)

> **When** I read GEMINI.md to understand why bkit locks `topicUpdateNarration: false`,
> **I want to** find an explicit justification matching the standard set by other locks (gemma/autoMemory/memoryManager),
> **so I can** trust that this is a deliberate engineering decision rather than an oversight.

**현재 상태**: GEMINI.md:41 영역에 4 잠금이 한 줄로 묶이고 M6만 정당화 부재.

**해결 (ENH-2)**: GEMINI.md:41 1줄 → 3줄 확장. M6 사유 명시 (Phase-Aware Context Engineering의 topic noise 회피).

### 3.3 JS-3 (ENH-3 매핑 문서)

> **When** I as a future bkit maintainer am tasked with integrating bkit `agent-memory` with Gemini CLI's Auto Memory (`extraction.patch`),
> **I want to** find a pre-written mapping document analyzing the JSON-vs-patch schemas + 5 integration scenarios with cost-benefit,
> **so I can** skip the discovery phase and start implementation in the next sprint.

**현재 상태**: bkit `lib/core/agent-memory.js`와 Gemini CLI `extraction.patch`(PR #26338) 사이 매핑 문서 부재.

**해결 (ENH-3)**: 신규 `docs/02-design/agent-memory-vs-extraction-patch-mapping.md` ~300-500 lines.

### 3.4 JS-4 (ENH-4 Auto Memory inbox 안내)

> **When** I use bkit and Gemini CLI together with Auto Memory enabled in my own settings (`autoMemory: true`),
> **I want to** be informed via SessionStart that the inbox exists and bkit's default posture is `false` (read-only awareness),
> **so I can** make an informed decision about whether to retain bkit's lock or override it personally.

**현재 상태**: bkit이 `autoMemory: false` 잠금. 사용자가 *수동 활성화* 시도해도 SessionStart에서 인지 없음.

**해결 (ENH-4, 조건부)**: session-start.js에 `hasAutoMemoryInbox` capability flag 조건부 안내 1줄. 안내 톤은 D-A2 결재.

### 3.5 JS-5 (ENH-5 capability flag)

> **When** Gemini CLI v0.43.0 stable releases (estimated 2026-06) with PR #25827 SessionStart `systemMessage` fix included,
> **I want to** have bkit automatically detect this and switch off the `BKIT_SESSION_START_VERBOSE` slim-default workaround,
> **so I can** receive full verbose SessionStart output without needing to set environment variables manually.

**현재 상태**: PR #25827은 main MERGED but v0.42.0 release 미포함. v0.43.0-preview.0 포함 확인. bkit은 *수동 detection* 부재.

**해결 (ENH-5)**: `lib/gemini/version.js`에 `hasSessionStartSystemMessageFix: isVersionAtLeast('0.43.0')` + session-start.js 조건부 자동 verbose 전환.

### 3.6 JS-6 (ENH-6 SKILL.md duplicate fix)

> **When** I as a Gemini CLI ecosystem developer reuse bkit's skills outside of bkit (e.g., copy `skills/btw/` to my own project),
> **I want to** find that bkit's SKILL.md YAML frontmatter follows the official standard without duplicate fields,
> **so I can** load the skill via Gemini CLI native skill resolver without parsing errors.

**현재 상태**: 6개 SKILL.md에 duplicate `classification:` YAML 위반 (E.2.2 grep 확정).

**해결 (ENH-6)**: 6개 파일의 후행 `classification:` 라인 삭제. `lib/skills/visibility.js` 참조로 정확한 값 결정.

```
ENH-6 (SKILL.md duplicate fix)
  → SKILL.md 43개 모두 Issue #15327 표준 부합

  → community contribution ↑
  → SKILL.md 표준 정합성 유지 trigger
```

### 8.3 Loop 3: v0.43.0 자동 전환 → 사용자 만족

```
ENH-3 (agentMemory ↔ extraction.patch 매핑 문서)

  → bkit cycle 가속화
```
