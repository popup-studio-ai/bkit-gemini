## §0. Executive Summary

### 0.1 Mission

bkit이 사용자에게 약속한 "21개 specialized agent"가 Gemini CLI 환경에서 **이름으로 직접 호출 가능**하도록 `LocalSubagentInvocation` registry에 등록한다. 현재 `gemini agents list` catalog는 21 agent를 인식하지만, `gemini -p "Use the <agent> agent..."` 호출 시 HTTP 404로 `generalist` fallback이 발생한다. 본 sprint는 이 갭을 완전히 해소하여 bkit이 광고하는 모든 agent가 실제로 의도된 역할(예: code-analyzer가 code quality 분석)을 수행하도록 보장한다.

### 0.2 Anti-Mission (본 sprint scope 외)

- bkit agent 인벤토리 변경 (21개 그대로 유지)
- agent 프롬프트/페르소나/도구 권한 변경 (`agents/*.md` frontmatter는 그대로)
- v2.1.0 Context Optimization 작업 (별도 `v2.1.0-context-optimization` sprint)
- 신규 v0.43.0 stable migration 작업 (별도 `v0.43.0-stable-migration` sprint)
- Gemini CLI fork 또는 upstream PR 제출 (필요시 carry forward)

### 0.3 4-Perspective Value

| 관점 | 가치 |
|---|---|
| **Problem** | 사용자가 `gemini -p "Use code-analyzer agent..."` 호출 시 `generalist` fallback이 실행되어, **광고한 specialized agent의 페르소나/도구 권한이 적용되지 않음**. fallback 응답이 정상적으로 보일 수 있어 사용자가 갭을 인지하기 어려운 silent quality degradation. |
| **Solution** | `gemini-extension.json` schema 분석 → `LocalSubagentInvocation` registry에 21 agent 등록 → dual-env (v0.39.1 + v0.42.0) 회귀 검증 → 모든 catalog/dispatch 경로 동기화. |
| **Function-UX** | 사용자가 `gemini -p "Use the qa-strategist agent to design test plan for ..."` 호출 시 qa-strategist의 정확한 프롬프트/도구 권한이 적용되어 응답. fallback 메시지 0건. |
| **Core Value** | "bkit이 명시한 21 agent = 실제 동작하는 21 agent" 1:1 contract. **silent fallback** → **explicit dispatch**. |

### 0.4 핵심 통계 (1 페이지)

|---|---|

| 위험도 | MEDIUM (gemini-extension.json schema 변경 / upstream Gemini CLI 동작 의존) |

## §1. Context Anchor

### 1.1 WHY (왜 진행하는가)

Sprint v0.42.0-stable-migration W4 실측 (2026-05-14):

```
[v0.39.1 + v0.42.0 동일 결과]
gemini -p "Use the code-analyzer agent to evaluate: var x = 1; eval(input);"
→ "code-analyzer agent was not found"
→ generalist fallback
→ stack trace: at async LocalSubagentInvocation.execute
   file:///.../@google/gemini-cli/bundle/chunk-7VVHSNDQ.js:309875:22
   code: 404
→ bkit Feature Usage: "Not Used: code-analyzer (Requested but entity not found)"
```

사용자 의도("`code-analyzer` agent로 분석")와 실제 동작(`generalist` fallback) 사이의 silent quality degradation. **사용자가 응답 품질 차이를 인지하기 어려움** = 더 위험.

### 1.2 WHO (이해관계자)

| 역할 | 영향 |
|---|---|
| bkit 사용자 (개발자) | "qa-strategist 호출하면 qa-strategist가 응답한다"는 mental model 회복 |
| bkit 사용자 (PM/CTO) | pm-discovery, pm-strategy 등 PM Agent Team 호출 시 정확한 페르소나 적용 |
| bkit 유지보수자 | Sprint v0.42.0+ 진행 시 dispatch 작동을 전제로 한 작업 (예: sprint-orchestrator의 specialist agent 위임) |
| Gemini CLI upstream | (잠재) bkit이 발견한 sub-agent registration schema 갭을 PR로 기여 가능 |

### 1.3 RISK

| ID | Risk | 가능성 | 영향 | 완화 |
|---|---|---|---|---|
| R-2.1.0-1 | gemini-extension.json schema가 agent registration을 지원하지 않음 (upstream 갭) | 중간 | 높음 | F1 research에서 schema 분석 → 미지원 시 R-2.1.0-2 대체 경로 |
| R-2.1.0-2 | LocalSubagentInvocation을 우회하는 별도 메커니즘 필요 (예: MCP tool 위장) | 낮음 | 중간 | F1 research에서 대체 경로 식별 |
| R-2.1.0-3 | bkit agent .md의 model/tools/temperature 필드가 LocalSubagentInvocation과 매핑되지 않음 | 중간 | 낮음 | F1 research에서 매핑 정합성 검증 |
| R-2.1.0-4 | dual-env 차이 (v0.39.1 spec ≠ v0.42.0 spec) | 낮음 | 중간 | F3 dual QA로 명시 검증 |
| R-2.1.0-5 | bkit이 의도한 21 agent ≠ Gemini CLI가 인식 가능한 agent 수 (등록 한계) | 매우 낮음 | 중간 | F1 research에서 한계 확인 |

## §6. Pre-mortem (사전 부검 시나리오)

### 시나리오 1: gemini-extension.json schema가 sub-agent 등록 미지원
- **확률**: 중간
- **원인**: Gemini CLI는 hooks/mcpServers/contextFileName/settings만 지원하고 sub-agent registration은 별도 API 부재
- **대응**: R-2.1.0-2 대체 경로 활성화 — MCP tool로 21 agent를 위장 또는 `agents register` CLI subcommand 사용

### 시나리오 2: LocalSubagentInvocation이 internal-only이라 외부 등록 불가
- **확률**: 낮음
- **대응**: Gemini CLI fork 또는 upstream PR 제출 carry forward

### 시나리오 3: dual-env 비대칭 (v0.39.1만 등록 가능, v0.42.0은 다른 schema)
- **확률**: 낮음
- **대응**: F1 research에서 미리 식별 → 두 schema 모두 지원하는 wrapper

### 시나리오 4: 21 agent 중 일부 (예: cto-lead, pdca-iterator)가 다른 agent를 Task spawn하는 패턴 (composite agent) 미지원
- **확률**: 중간
- **대응**: 단순 agent (code-analyzer, gap-detector 등) 먼저 등록 + composite agent는 별도 단계

### 시나리오 5: registration 변경이 `gemini agents list` catalog 변동 유발
- **확률**: 낮음
- **대응**: F3.6에서 명시 검증 — variance 발견 시 rollback
