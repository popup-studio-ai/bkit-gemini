## §1. Sprint Overview

### 1.1 목표

Gemini CLI Extension인 bkit-gemini에 **L3 Full-Auto / L4 Autonomous** 자동화 수준을 도입하여, 후속 sprint(S1~S6)를 사용자 결재 없이 안전하게 진행 가능하게 한다. 임시방편 거부 원칙에 따라 **Track A(bkit Extension 자체 강화) + Track B(Gemini CLI 본체 hook contract PR)** 2-track 병행으로 근본 해결한다.

### 1.2 부트스트랩 Paradox 명시

S7은 **자기 자신을 가능하게 하는 sprint**이므로 S7 자체는 L2 한도에서 진행한다(L4 적용 불가). PDCA phase 전환마다 사용자 결재 필요. 단, S7 완료 후 `/control level L4` 실행 시 S1~S6는 모두 L4로 자동 진행 가능해진다.

### 1.3 Sprint Scope

**In scope**:
- bkit `hooks/scripts/before-tool.js`, `after-tool.js` 강화 (L4 자동 승인 로직)
- 신규 `hooks/scripts/before-phase.js` (PDCA phase 전환 자동화)
- 신규 `lib/core/trust-score.js` (동적 trust score 알고리즘)
- `skills/control/SKILL.md` L3/L4 unlock + Gemini-specific 명세
- `.bkit/runtime/control-state.json` 스키마 확장 (v1.0 → v1.1)
- `lib/gemini/policy.js` destructive op deny 매트릭스 강화
- L4 안전 가드: hard deny, auto checkpoint, rollback, audit log
- Gemini CLI 본체 hook contract 확장 PR (Track B)
- 회귀 testcase: `tests/regression/automation/*.test.js` (L4 시나리오 10건)
- RCA 문서: `docs/03-analysis/v2.0.7-l4-automation-rca.md`

**Out of scope** (다른 sprint에서 처리):
- 21 agent dispatch (S3에서 처리)
- Context Engineering 4-strategy (S5에서 처리)
- Baseline test fix (S2, S6에서 처리)
- README/QUICKSTART 분할 (S4에서 처리)
- v2.1.0+ feature (별도 minor)

## §3. Key Decisions (D1~D14)

### D1: 2-Track 병행 (Track A + Track B)
- **결정**: Track A(bkit 강화)는 즉시 가치 + Track B(본체 PR)는 항구 해결. 두 track 모두 진행.
- **근거**: 사용자 명시 "임시방편 거부" — Track A 단독은 임시방편이지만 Track B와 짝이면 근본.
- **대안**: Track A 단독(임시방편), Track B 단독(머지 거부 시 위험). 모두 reject.

### D2: 부트스트랩 — S7 자체는 L2로 진행
- **결정**: S7 PDCA phase 전환마다 사용자 결재 필요 (L4 적용 불가).
- **근거**: 자기 자신을 enable하는 sprint이므로 순환 의존.
- **대안**: L4 강제 모드(unsafe), 별도 minor로 분리(scope 위반). 모두 reject.

### D3: trust_score 알고리즘
- **결정**: success +5, rejection -10, cap [0, 100], cooldown 100ms boundary, L4 entry @ 80, L2 downgrade @ 5건 연속 rejection.
- **근거**: 비대칭(+5/-10)으로 신뢰 회복 느림 + 빠른 강등. 안전 우선.
- **대안**: 대칭 (+5/-5, oscillation 위험), exponential decay (복잡도 증가). 모두 reject.

### D4: Hook 진입점 = `before-tool.js`
- **결정**: tool 호출 직전이 자동 승인의 가장 효과적 시점.
- **근거**: Gemini CLI의 hook spec 분석 (Wave 0 RCA에서 검증).
- **대안**: before-model (너무 추상적), after-tool (이미 실행됨). 모두 reject.

### D5: tool whitelist는 read-only ops 기본 포함
- **결정**: Read, Glob, Grep, LSP, list_directory 등 read-only는 L4 시 자동 'allow'.
- **근거**: M3 게이트 (보안 정책) 위반 위험 0. 실수해도 데이터 손상 없음.
- **대안**: 모든 tool 자동 'allow' (위험), 화이트리스트 없음 (자동화 불가). reject.

### D6: Destructive ops 영구 deny (L4 무관)
- **결정**: rm -rf, force-push, drop table 등 destructive는 L4에서도 무조건 deny.
- **근거**: 사용자 명시 "꼼꼼하고 완벽하게 + 임시방편 X". 자동화 = 안전성 절대 보장.
- **대안**: L4에서 destructive도 자동 'allow' (절대 reject), 사용자별 화이트리스트 (복잡).

### D7: AST 기반 명령 파싱 (heredoc/escape 우회 차단) + D16 라이브러리
- **결정**: `lib/core/cmd-parser.js`로 명령을 AST로 파싱한 후 destructive 검출. **D16 결정에 따라 hybrid 전략 채택**.
- **근거**: 단순 정규식은 `rm -rf $X`, heredoc, `\` escape, base64 인코딩 등 우회 가능.
- **대안**: 정규식 only (R7-E 위험), shell 실제 실행 후 검증 (이미 실행됨). reject.

### D16' (revised, Wave 0 Day 1): AST 라이브러리 = 자체 mini-parser only
- **결정 (2026-05-14 RCA 후 revision)**: `lib/core/cmd-parser.js` 자체 mini-parser **단독** (regex + token + escape unwind). 외부 의존성 0. v2.1.0에서 정밀 AST 검증 필요 시 재평가.
- **근거 (Wave 0 RCA §4)**:
  - `bash-parser` (vorpaljs/bash-parser, npm): 마지막 publish **9년 전 (2017)**, abandoned 상태 → bkit production 신뢰 부적합 (사용자 명시 "꼼꼼하게" 원칙 위반)
  - `@ericcornelissen/bash-parser` (fork): 활성 추정이나 v2.0.7 prerequisite scope에서 외부 의존성 추가는 부적절 → v2.1.0 재평가

## §6. trust_score 알고리즘 상세 (D3 구현)

### 6.1 TrustScoreManager 인터페이스

```js
class TrustScoreManager {
  constructor(projectDir) { /* control-state.json 로드 */ }

  getScore() { return this.state.trustScore; }
  getLevel() { return this.state.currentLevel; }

  recordDecision(decision) {
    // decision: { type: 'allow'|'deny'|'rejection', tool, timestamp, ms }
    this.state.recentDecisions.push(decision);
    if (this.state.recentDecisions.length > 50) this.state.recentDecisions.shift();

    if (decision.type === 'allow' && !decision.rejected) {
      this.state.trustScore = Math.min(100, this.state.trustScore + 5);
    } else if (decision.type === 'rejection') {
      this.state.trustScore = Math.max(0, this.state.trustScore - 10);
    }

    this.checkAutoDowngrade();
    this.persist();
  }

  canEscalate(targetLevel) {
    if (targetLevel <= this.state.currentLevel) return true;
    if (this.state.cooldownUntil && Date.now() < this.state.cooldownUntil) return false;
    if (targetLevel === 4) {
      return this.state.trustScore >= 80
        && this.countRecentRejections(10) === 0;
    }
    if (targetLevel === 3) {
      return this.state.trustScore >= 60
        && this.countRecentRejections(5) <= 1;
    }
    return false;
  }

  checkAutoDowngrade() {
    const recent5 = this.state.recentDecisions.slice(-5);
    const allRejection = recent5.length === 5 && recent5.every(d => d.type === 'rejection');
    if (allRejection && this.state.currentLevel > 2) {
      this.downgrade(2, 'auto: 5 consecutive rejections');
    }
  }

  downgrade(targetLevel, reason) {
    this.state.previousLevel = this.state.currentLevel;
    this.state.currentLevel = targetLevel;
    this.state.lastDowngrade = new Date().toISOString();
    this.state.levelChangeReason = reason;
    this.state.cooldownUntil = Date.now() + 24 * 60 * 60 * 1000; // 24h cooldown
    this.state.automationHistory.push({
      from: this.state.previousLevel,
      to: targetLevel,
      reason,
      timestamp: this.state.lastDowngrade
    });
    this.persist();
  }
  // ... persist(), countRecentRejections(), etc.
}
```

### 6.2 100ms boundary cooldown

연속 결정 사이가 100ms 이내인 경우 같은 의도로 간주(잡음 무시):

```js
shouldRecord(decision) {
  const last = this.state.recentDecisions.slice(-1)[0];
  if (last && (decision.timestamp - last.timestamp) < 100) return false;
  return true;
}
```
