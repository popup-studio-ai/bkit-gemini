#### 3.1.3 `dispatchViaSpawnAgent(match, mcpClient)` — Public API

```js
/**
 * Dispatch matched agent via MCP spawn_agent tool.
 *
 * @param {DispatchMatch} match - Result from matchDispatchIntent
 * @param {MCPClient} mcpClient - MCP server client (injected for test)
 * @returns {Promise<DispatchResult>}
 */
async function dispatchViaSpawnAgent(match, mcpClient) {
  const agentMeta = AGENTS[match.agentName];  // from mcp/bkit-server.js or local cache
  const timeout = (agentMeta.timeout_mins || 5) * 60 * 1000;

  const result = await mcpClient.call('spawn_agent', {
    agent_name: match.agentName,
    task: match.task,
    context: {
      dispatch_source: 'bkit-hook',
      user_prompt_hash: hashPrompt(match.task),
      language: match.language,
    },
    timeout,
  });

  return {
    success: true,
    agentName: match.agentName,
    output: result.content,
    elapsedMs: result.elapsedMs,
  };
}
```

**Returns** (`DispatchResult`):
```ts
type DispatchResult = {
  success: boolean;
  agentName: string;
  output: string;
  elapsedMs: number;
  error?: string;  // present if !success
}
```

**FR 매핑**: FR-5, FR-6.

#### 3.1.5 `shouldUseTrackABypass(env, version)` — Capability Logic

```js
/**
 * Decide whether Track A bypass should be active.
 *
 * @param {string} env - BKIT_NATIVE_AGENT_DISPATCH value (auto|on|off|ask)
 * @param {object} version - lib/gemini/version.js capability output
 * @returns {{ active: boolean, mode: 'auto'|'on'|'off'|'ask', reason: string }}
 */
function shouldUseTrackABypass(env, version) {
  const mode = env || 'auto';
  if (mode === 'off') return { active: false, mode, reason: 'explicit off' };
  if (mode === 'on') return { active: true, mode, reason: 'explicit on' };
  if (mode === 'ask') return { active: true, mode, reason: 'ask mode (confirmation needed)' };
  // mode === 'auto'
  if (version.hasNativeAgentDispatch) {
    return { active: false, mode, reason: 'auto: native dispatch available (v0.44.0+)' };
  }
  return { active: true, mode, reason: 'auto: native dispatch unavailable' };
}
```

**FR 매핑**: FR-8, FR-9. **AC 매핑**: AC-T1, AC-T2, AC-T3, AC-T4.

function tryAgentDispatch(prompt, projectDir) {
  try {
    const { matchDispatchIntent, dispatchViaSpawnAgent, wrapWithBoundary,
            shouldUseTrackABypass, loadDispatchMemory } = require(path.join(libPath, 'gemini', 'agent-dispatch'));
    const { getCapabilities } = require(path.join(libPath, 'gemini', 'version'));

    const memory = loadDispatchMemory(projectDir);
    if (decision.mode === 'ask' || memory.confirmed_count < 2) {

    }

    const result = dispatchViaSpawnAgentSync(match);  // sync wrapper for hook
    if (!result.success) {

      logDispatchFailure(projectDir, match, result.error);
      return null;

      agentName: match.agentName,

## §5. Hook Integration

### 5.1 Hook Pipeline 위치

```
SessionStart
   ↓
BeforeAgent (intent detection — 기존 동작)
   ↓
BeforeModel ★ (이 sprint 의 진입점)
   │
   ├─ tryAgentDispatch() — Track A 인터셉트 분기 (신규)
   │   │
   │   ├─ 매치 + decision.active → spawn_agent + boundary wrap → return 'replace'
   │   ├─ 매치 + decision.mode==='ask' → AskUserQuestion → confirm 후 dispatch
   │   ├─ 매치 안 됨 → null (다음 단계로)
   │   └─ 에러 → null (silent fallback)
   │
   └─ 기존 PDCA Context Anchor + Model Routing 로직 (변경 없음)
   ↓
Model invocation
   │
   ├─ 'replace' decision → 모델 호출 생략, replacedPrompt 가 응답으로
   └─ 'allow' decision → Gemini CLI 본체로 위임 (LocalSubagentInvocation 등)
   ↓
AfterModel
```

### 5.2 Hook Decision Type 확장

기존 `before-model.js` 의 반환은 `{decision: 'allow', additionalContext?: string}` 였음. 본 sprint 에서 새로운 decision type `'replace'` 추가:

```js
type HookResult =
  | { decision: 'allow'; additionalContext?: string }       // 기존
  | { decision: 'replace'; replacedPrompt: string; meta?: object };  // 신규
```

Gemini CLI runtime hook SDK 가 `'replace'` decision 지원 여부 확인 필요 (Track B RCA 시 동시 확인). 미지원 시 대안:
- 사용자 prompt 를 그대로 두되 `additionalContext` 에 agent 응답 주입 + 모델에 "Return the additionalContext as-is" 지시 (degraded mode)
- 또는 stdout 에 직접 출력 후 process.exit(0) (legacy command mode)

**Note**: Track B 의 minimal reproducer 작성 시 hook SDK 의 'replace' 지원 여부 동시 확인 — RCA doc 에 반영.

| **L4 Validation** | `META_PATTERNS` 거부 + `AGENT_WHITELIST` exact match | agent_name 이 21개 화이트리스트 내 OR null | unit test (whitelist-only.test.js) |
| **L5 DB (Catalog)** | `agents/<name>.md` frontmatter + AGENTS object in `mcp/bkit-server.js` | name/model/tools/temperature/timeout_mins 모두 존재 | catalog dump test |
| **L6 Response (Execution)** | MCP spawn_agent → agent 실행 → output 반환 | output 가 string 타입, empty 아님 | regression suite |
| **L7 Client (Wrap)** | `wrapWithBoundary()` → `[Agent: X]\n...\n[End Agent Output]` | boundary 라벨 형식 정확, main context leak 0 | boundary-wrap.test.js + AC-I2 수동 검증 |
| **L8 UI (Output)** | Gemini CLI stdout → 사용자 터미널 | boundary 라벨 사용자 가시 + 응답 truncation 0 | manual UX test |

### 6.1 s1Score 계산

```
s1Score = (passedLayers / totalLayers) * 100
       = (8 / 8) * 100
       = 100
```

각 layer 별 PASS 기준:
- L1 PASS: prompt 가 raw 로 매처 도달 — 중간 sanitize 단계 0건 검증
- L2 PASS: hook input 의 schema 가 `{prompt: string, projectDir: string, ...}` 형식
- L3 PASS: `agent-dispatch.js` exports 가 Design §3.1 시그니처와 일치
- L4 PASS: META_PATTERNS / WHITELIST 모두 reject path 검증 (unit test)
- L5 PASS: AGENTS object 의 21개 agent 모두 frontmatter 5 필수 키 보유
- L6 PASS: 21 agent × 1 sample = 21 spawn 성공
- L7 PASS: 모든 응답이 `[Agent: ...]` 시작 + `[End Agent Output]` 종료
- L8 PASS: 사용자 manual UX test 통과 (Final Report §10.x)

### 6.2 Anti-leak 검증 (AC-I2)

**검증 절차**:
1. dispatch 시도 직전 main context snapshot (size, hash)
2. agent 응답 후 main context snapshot
3. diff = (응답 후 - 응답 전) — 이 diff 의 모든 추가 텍스트는 `[Agent: ...]` ~ `[End Agent Output]` 안에 위치해야 함
4. agent .md 의 frontmatter 텍스트 (예: "You are code-analyzer...") 가 main context 에 0건

## §8. False-Positive Defense Algorithm

### 8.1 Algorithm

```
INPUT: prompt (string), projectDir (string)
OUTPUT: { proceed: boolean, askUser: boolean, match: DispatchMatch | null }

1. memory = loadDispatchMemory(projectDir)

2. match = matchDispatchIntent(prompt)
3. if !match → return { proceed: false, askUser: false, match: null }

4. // Layer 1 defense: meta pattern
   if META_PATTERNS[match.language].test(prompt):
      return { proceed: false, askUser: false, match: null }

5. // Layer 2 defense: code-block check
   if isInsideCodeBlock(prompt, match.agentName):
      return { proceed: false, askUser: false, match: null }

6. // Layer 3 defense: confirmation history
   if memory.confirmed_count < 2 OR env === 'ask':
      return { proceed: true, askUser: true, match }
   else:
      return { proceed: true, askUser: false, match }
```

### 8.2 `isInsideCodeBlock(prompt, agentName)`

```js
function isInsideCodeBlock(prompt, agentName) {
  // 백틱 ``` ... ``` 블록 안인지 검사
  const blocks = prompt.match(/```[\s\S]*?```/g) || [];
  return blocks.some(block => block.includes(agentName));
}
```

### 8.3 AskUserQuestion 통합

Gemini CLI 의 user-confirmation 메커니즘 활용. Track B 의 RCA 시점에 정확한 API 확인 — 가정:
- `process.stderr.write('Dispatch <agent> agent for: <task>? [Y/n] ')` + stdin read (legacy mode)
- 또는 hook SDK 의 `decision: 'confirm', question: '...'` (v0.31.0+ SDK)

## §14. Performance Design

### 14.1 Latency Budget (NFR-1, NFR-2)

| 단계 | 예산 |
|---|---|
| matchDispatchIntent (regex) | < 10ms |
| isInsideCodeBlock | < 2ms |
| loadDispatchMemory | < 5ms (small JSON file) |
| spawn_agent MCP setup | < 50ms (MCP protocol roundtrip) |
| agent execution | varies (agent 자체 — 사용자가 의도한 비용) |
| wrapWithBoundary | < 1ms |
| **Total overhead** | **< 100ms** (NFR-2) |

### 14.2 Regex Compile Cache

`DISPATCH_PATTERNS` 와 `META_PATTERNS` 는 module 로드 시 1회 컴파일 (정규식 리터럴이라 자동). Hot path 에서 추가 컴파일 없음.

### 14.3 Memory Footprint (NFR-3)

- DISPATCH_PATTERNS / META_PATTERNS module: ~10KB
- DispatchMemory cache: 단일 dispatch 당 ~1KB (prompt hash + agent name + timestamp)
- 1000 dispatch / day → < 1MB

## §15. Security Design

### 15.1 우회 정책 우회 방지

Track A 가 MCP `spawn_agent` 호출 → `mcp/bkit-server.js` 의 기존 정책 엔진 (`lib/gemini/policy.js`) 통과. agent 가 deny 패턴에 명시된 도구 사용 시 거부 — Track A 가 정책을 우회하지 않음.

**M3 보안 정책**: PASS.

### 15.2 prompt injection 방어

- `matchDispatchIntent` 가 raw prompt 를 정규식만 적용 — eval / exec 없음
- agent_name 은 21개 화이트리스트 exact match — injection 방지
- task 는 spawn_agent 의 `task` 인자로 전달 — agent 가 내부적으로 sanitize

### 15.3 audit trail

모든 dispatch 가 audit log 에 기록 — 정책 위반 forensic 가능.
