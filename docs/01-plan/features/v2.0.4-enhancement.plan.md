## Executive Summary

### 전략 전환: Extension 제약 → MCP Server 자유

이전 계획(v2.0.4 Option C)은 Gemini CLI Extension을 "관찰자 + 조언자"로 규정하고,
Skill 프롬프트 가이드 + Hook 인라인이라는 **수동적** 접근을 선택했다.

**핵심 발견**: MCP (Model Context Protocol) Server는 Extension의 구조적 제약을 우회하는 escape hatch이다.

MCP Server는 **별도의 Node.js 프로세스**로 실행되며:
- `child_process`로 임의 shell 명령 실행 (jest, playwright, curl 등)
- `fs`로 파일 읽기/쓰기 (PDCA 문서, audit 로그)
- `http`로 외부 API 호출
- tool call 간 상태 유지 (인메모리 state)
- 구조화된 결과를 main agent에 반환

bkit-gemini은 이미 `mcp/spawn-agent-server.js` (907줄)를 보유하고 있으나,
`.gemini/settings.json`에 MCP server로 **등록되어 있지 않다**.
등록하면 Gemini agent가 커스텀 도구처럼 MCP tool을 호출할 수 있다.

### Before / After 비교

| 항목 | 이전 계획 (Option C) | 현재 계획 (MCP-Centric) |
|------|---------------------|------------------------|
| 핵심 전략 | Skill 프롬프트 + Hook 인라인 | **MCP Server = 오케스트레이션 엔진** |
| PM 팀 실행 | spawn_agent 순차 (프롬프트 지시) | **bkit_pm_run** MCP tool (내부 자동 실행) |
| QA 테스트 | qa-monitor shell 가이드 | **bkit_qa_run** MCP tool (jest/playwright 자동) |
| PDCA 반복 | pdca-iterator 에이전트 (프롬프트) | **bkit_iterate** MCP tool (내부 gap 분석 루프) |
| Audit 기록 | after-tool.js 인라인 10줄 | after-tool.js 인라인 + **bkit_audit_query** 검색 tool |
| Checkpoint | 프롬프트 "확인 요청" 지시 | **bkit_checkpoint** MCP tool (create/restore) |
| UX Parity | 80% | **90%** |
| 자동화 수준 | 낮음~중간 | **높음** |
| 공수 | 30-40h | **30-38h** (MCP는 기존 코드 활용) |

## 1. 의도 분석 (Intent Discovery)

### 1.1 WHY: 이 고도화가 필요한 이유

**표면적 목표**: bkit-claude-code와의 기능 동등성 확보

**실제 목표**: bkit-claude-code 사용자가 경험하는 **체계적인 PDCA 워크플로우 UX**를
Gemini CLI 사용자에게도 제공하는 것.

**구체적 UX 격차 (변함없음)**:
- CC 사용자: `/pdca plan` → Context Anchor 자동 생성 → `/pdca design` → 3가지 아키텍처 옵션 + 사용자 선택 → `/pdca do --scope module-1` → Decision Record Chain 표시 + 범위 필터링
- Gemini 사용자: `/pdca plan` → 문서 생성 → `/pdca design` → 옵션 없음 → `/pdca do` → 범위 제어 없음

### 1.2 새로운 접근: MCP Server as Orchestration Engine

이전 계획은 Gemini CLI Extension의 한계를 받아들이고 "프롬프트 가이드"에 의존했다.
하지만 MCP Server를 활용하면 **Extension이 할 수 없는 것을 MCP가 대신 할 수 있다**.

```
Extension 제약                          MCP Server로 우회
─────────────────────────────────────   ────────────────────────────────────
❌ Skills에서 코드 실행 불가             ✅ MCP tool 내부에서 코드 실행 가능
❌ Hook 간 직접 통신 불가               ✅ MCP Server 프로세스 내 state 공유
❌ 순차 에이전트만 (병렬 불가)           ✅ MCP 내부에서 child_process 병렬 가능
❌ Hook 동기 실행 (60초 타임아웃)        ✅ MCP tool은 비동기, 장시간 실행 가능
❌ stdout JSON만 반환                    ✅ MCP tool은 구조화된 결과 반환
❌ CLI 내부 상태 접근 불가               ✅ MCP Server 자체 state 관리
```

### 1.3 WHY MCP가 구조적 제약을 극복하는가

[gemini-cli-extension-constraints.md](../research/gemini-cli-extension-constraints.md) 문서에서
식별된 6가지 CRITICAL/HIGH 제약 중 **4가지를 MCP Server가 해결**한다:

| Constraint Document 항목 | 심각도 | MCP 해결 여부 |
|--------------------------|--------|--------------|
| Hook System 한계 (이벤트 부족) | CRITICAL | **부분 해결** — MCP tool이 오케스트레이션 로직을 내부에서 처리하므로 SubagentStart/Stop 이벤트가 불필요 |
| Agent System 한계 (팀/병렬 불가) | CRITICAL | **해결** — bkit_team_run MCP tool이 내부에서 순차 조율. 외부에서는 단일 tool call |
| Skill System 한계 (코드 실행 불가) | HIGH | **해결** — MCP tool이 코드 실행 주체. Skill은 MCP tool 호출 가이드 역할 |
| Context Engineering 한계 (토큰 관리 불가) | HIGH | **미해결** — MCP Server도 CLI 토큰에 접근 불가 |
| Policy/Automation 한계 (allow 불가) | MEDIUM | **미해결** — MCP와 무관한 정책 엔진 영역 |
| Extension Packaging 한계 | MEDIUM | **해결** — MCP Server는 별도 프로세스이므로 패키징 제약 없음 |

### 1.4 전략적 결론

> **"MCP Server = bkit의 실행 엔진, Skill = 사용자 인터페이스, Hook = 관찰 레이어"**

이 3계층 분리가 Gemini CLI에서 달성 가능한 최대 자동화 수준이다.

## 2. Architecture: MCP Server-Centric Design

### 2.1 전체 아키텍처

```
┌─────────────────────────────────────────────────────────────────┐
│  Gemini CLI                                                      │
│                                                                  │
│  User → Skill (pdca) → agent calls MCP tool                      │
│                           │                                      │
│  ┌────────────────────────┴────────────────────────────────┐     │
│  │  bkit MCP Server (Node.js process)                       │     │
│  │                                                          │     │
│  │  Tools:                                                  │     │
│  │  ├─ spawn_agent (EXISTING — agent 위임)                   │     │
│  │  ├─ list_agents (EXISTING — agent 목록)                   │     │
│  │  ├─ get_agent_info (EXISTING — agent 상세)                │     │
│  │  ├─ team_create (EXISTING — team 생성)                    │     │
│  │  ├─ team_assign (EXISTING — team 작업 할당)               │     │
│  │  ├─ team_status (EXISTING — team 상태)                    │     │
│  │  ├─ bkit_pm_run (NEW — PM pipeline 자동 실행)             │     │
│  │  ├─ bkit_qa_run (NEW — QA L1-L5 테스트 실행)             │     │
│  │  ├─ bkit_iterate (NEW — gap+fix+recheck 반복 루프)        │     │
│  │  ├─ bkit_team_run (NEW — 순차 오케스트레이션)              │     │
│  │  ├─ bkit_audit_query (NEW — audit 로그 검색)              │     │
│  │  ├─ bkit_checkpoint (NEW — create/restore 체크포인트)      │     │
│  │  └─ bkit_pdca_status (NEW — PDCA 상태 조회/업데이트)       │     │
│  │                                                          │     │
│  │  Internal capabilities (Node.js full runtime):           │     │
│  │  ├─ child_process.spawn → jest, playwright, curl 실행     │     │
│  │  ├─ fs → PDCA 문서 읽기/쓰기, audit JSONL                │     │
│  │  ├─ Sequential agent coordination (내부 상태 머신)         │     │
│  │  ├─ Quality gate evaluation (통과/실패 판정)              │     │
│  │  └─ In-memory state (tool call 간 상태 유지)              │     │
│  └──────────────────────────────────────────────────────────┘     │
│                                                                  │
│  Hooks (minimal, observation only):                              │
│  ├─ SessionStart → PDCA state 초기화, GEMINI.md context 주입      │
│  ├─ AfterTool → audit trail JSONL 기록 (인라인 10줄)              │
│  ├─ BeforeTool → guardrail 패턴 체크 (기존 유지)                  │
│  ├─ BeforeAgent → intent classification (lib/intent 연결)        │
│  └─ SessionEnd → audit flush (best-effort)                       │
│                                                                  │
│  Skills (user interface, prompt guides):                         │
│  ├─ pdca/ → MCP tool 호출 시점 가이드 + Context Anchor 지시       │
│  ├─ plan-plus/ → YAGNI review + bkit_pm_run 호출 가이드           │
│  ├─ qa-phase/ → bkit_qa_run 호출 가이드                          │
│  └─ 8 empty skills → SKILL.md 작성                               │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 MCP Server 등록

현재 `.gemini/settings.json`:
```json
{
  "experimental": {
    "enableAgents": true
  }
}
```

변경 후:
```json
{
  "experimental": {
    "enableAgents": true
  },
  "mcpServers": {
    "bkit-agents": {
      "command": "node",
      "args": ["mcp/spawn-agent-server.js"],
      "cwd": "${workspaceFolder}",
      "timeout": 120000
    }
  }
}
```

> **참고**: `mcp/spawn-agent-server.js`는 이미 907줄의 MCP Server 구현체이지만,
> `.gemini/settings.json`에 등록되어 있지 않아 Gemini agent가 호출할 수 없었다.
> 등록만 하면 기존 `spawn_agent`, `list_agents`, `team_create` 등이 즉시 사용 가능.

### 2.3 3계층 역할 분리

| 계층 | 역할 | 실행 환경 | 제약 |
|------|------|----------|------|
| **MCP Server** | 오케스트레이션, 코드 실행, 상태 관리 | 별도 Node.js 프로세스 | JSON-RPC 통신, Gemini agent가 호출해야만 실행 |
| **Skills** | 사용자 인터페이스, 워크플로우 가이드, MCP tool 호출 시점 안내 | LLM 프롬프트 주입 | 코드 실행 불가, 프롬프트로만 행동 유도 |
| **Hooks** | 관찰, 기록, 차단 (guardrail) | Hook subprocess | 동기 실행, 60초 타임아웃, JSON stdout only |

**핵심 원칙**: Hooks는 가볍게 (관찰+기록), Skills는 똑똑하게 (가이드+판단), MCP는 강하게 (실행+조율).

## 3. MCP Tool Specifications

### 3.1 bkit_pm_run — PM Pipeline 자동 실행

**CC 동등 기능**: pm-lead가 Task()로 pm-discovery, pm-strategy, pm-research, pm-prd 4개 에이전트를 병렬 스폰

**Gemini MCP 구현**: 단일 MCP tool call로 PM 분석 파이프라인을 내부에서 순차 실행

```
Tool Name: bkit_pm_run
Description: Run the PM analysis pipeline — discovery, strategy, research, PRD generation
```

**Input Schema**:
```json
{
  "type": "object",
  "properties": {
    "feature": {
      "type": "string",
      "description": "Feature name (e.g., 'user-auth', 'payment-v2')"
    },
    "phases": {
      "type": "array",
      "items": {
        "type": "string",
        "enum": ["discovery", "strategy", "research", "prd"]
      },
      "description": "PM phases to execute (default: all 4)",
      "default": ["discovery", "strategy", "research", "prd"]
    },
    "projectContext": {
      "type": "string",
      "description": "Additional project context (optional)"
    }
  },
  "required": ["feature"]
}
```

**Internal Logic** (spawn-agent-server.js에 추가):
```javascript
async handleBkitPmRun(args) {
  const { feature, phases = ['discovery', 'strategy', 'research', 'prd'] } = args;
  const results = {};
  const projectDir = process.cwd();
  const pmDir = path.join(projectDir, 'docs', '00-pm');
  
  // Phase 1: Read project context
  const packageJson = this.readFileIfExists(path.join(projectDir, 'package.json'));
  const geminiMd = this.readFileIfExists(path.join(projectDir, 'GEMINI.md'));
  const context = { packageJson, geminiMd, feature };

  // Phase 2: Sequential PM analysis (CC에서는 병렬이지만 MCP 내부에서 순차)
  if (phases.includes('discovery')) {
    results.discovery = await this.runAgentInternal('pm-discovery', 
      `Analyze feature "${feature}" — Opportunity Solution Tree`, context);
  }
  if (phases.includes('strategy')) {
    results.strategy = await this.runAgentInternal('pm-strategy',
      `Create Value Proposition + Lean Canvas for "${feature}"`, 
      { ...context, discoveryResult: results.discovery });
  }
  if (phases.includes('research')) {
    results.research = await this.runAgentInternal('pm-research',
      `Generate Personas + Competitors + TAM/SAM/SOM for "${feature}"`,
      { ...context, strategyResult: results.strategy });
  }

  // Phase 3: PRD Synthesis
  if (phases.includes('prd')) {
    const prdContent = await this.runAgentInternal('pm-prd',
      `Synthesize PRD from analysis results for "${feature}"`,
      { ...context, ...results });
    
    // Write PRD file
    fs.mkdirSync(pmDir, { recursive: true });
    const prdPath = path.join(pmDir, `${feature}.prd.md`);
    fs.writeFileSync(prdPath, prdContent.output || '');
    results.prd = { path: prdPath, content: prdContent };
  }

  return {
    content: [{
      type: 'text',
      text: JSON.stringify({
        status: 'complete',
        feature,
        phasesExecuted: phases,
        prdPath: results.prd?.path || null,
        summary: this.extractSummary(results),
        personas: results.research?.personas || [],
        competitors: results.research?.competitors || []
      }, null, 2)
    }]
  };
}
```

**Output Schema**:
```json
{
  "status": "complete",
  "feature": "user-auth",
  "phasesExecuted": ["discovery", "strategy", "research", "prd"],
  "prdPath": "docs/00-pm/user-auth.prd.md",
  "summary": "...",
  "personas": [...],
  "competitors": [...]
}
```

**UX Parity**: 90% — CC는 4개 에이전트를 병렬로 스폰하지만,
MCP 내부 순차 실행도 사용자 입장에서는 단일 명령으로 PRD가 생성되므로 체감 UX는 동등.

### 3.2 bkit_qa_run — QA L1-L5 테스트 실행

**CC 동등 기능**: qa-lead가 5개 QA agent를 오케스트레이션 + Chrome MCP로 L3-L5 브라우저 테스트

**Gemini MCP 구현**: MCP tool 내부에서 child_process로 테스트 프레임워크 직접 실행

```
Tool Name: bkit_qa_run
Description: Run QA tests at specified levels — L1 unit, L2 API, L3-L5 E2E (if available)
```

**Input Schema**:
```json
{
  "type": "object",
  "properties": {
    "feature": {
      "type": "string",
      "description": "Feature name to test"
    },
    "levels": {
      "type": "array",
      "items": { "type": "integer", "enum": [1, 2, 3, 4, 5] },
      "description": "Test levels to execute (default: [1, 2])",
      "default": [1, 2]
    },
    "testDir": {
      "type": "string",
      "description": "Test directory path (default: auto-detect)"
    },
    "testPattern": {
      "type": "string",
      "description": "Test file pattern (e.g., '**/*.test.js')"
    }
  },
  "required": ["feature"]
}
```

**Internal Logic**:
```javascript
async handleBkitQaRun(args) {
  const { feature, levels = [1, 2], testDir, testPattern } = args;
  const projectDir = process.cwd();
  const results = { passRate: 0, coverage: 0, defects: [], levelResults: {} };

  // L1: Unit Tests
  if (levels.includes(1)) {
    const testFramework = this.detectTestFramework(projectDir);
    // testFramework: 'jest' | 'vitest' | 'node-test' | null
    
    if (testFramework) {
      const cmd = this.getTestCommand(testFramework, testDir, testPattern);
      // e.g., 'npx jest --json --testPathPattern=feature'
      const l1Result = await this.execCommand(cmd, { cwd: projectDir, timeout: 120000 });
      results.levelResults.L1 = this.parseTestOutput(testFramework, l1Result);
    } else {
      results.levelResults.L1 = { status: 'skipped', reason: 'No test framework detected' };
    }
  }

  // L2: API Endpoint Tests
  if (levels.includes(2)) {
    const endpoints = this.detectApiEndpoints(projectDir, feature);
    const l2Results = [];
    
    for (const endpoint of endpoints) {
      const curlResult = await this.execCommand(
        `curl -s -o /dev/null -w '%{http_code}' ${endpoint.url}`,
        { cwd: projectDir, timeout: 10000 }
      );
      l2Results.push({
        endpoint: endpoint.url,
        method: endpoint.method,
        statusCode: parseInt(curlResult.stdout),
        pass: parseInt(curlResult.stdout) < 400
      });
    }
    results.levelResults.L2 = { tests: l2Results, passRate: this.calcPassRate(l2Results) };
  }

  // L3-L5: E2E Tests (Playwright / Cypress if installed)
  if (levels.some(l => l >= 3)) {
    const e2eFramework = this.detectE2eFramework(projectDir);
    // e2eFramework: 'playwright' | 'cypress' | null
    
    if (e2eFramework === 'playwright') {
      const cmd = `npx playwright test --reporter=json ${testPattern || ''}`;
      const e2eResult = await this.execCommand(cmd, { cwd: projectDir, timeout: 300000 });
      results.levelResults.L3_L5 = this.parsePlaywrightOutput(e2eResult);
    } else if (e2eFramework === 'cypress') {
      const cmd = `npx cypress run --reporter json ${testPattern || ''}`;
      const e2eResult = await this.execCommand(cmd, { cwd: projectDir, timeout: 300000 });
      results.levelResults.L3_L5 = this.parseCypressOutput(e2eResult);
    } else {
      results.levelResults.L3_L5 = {
        status: 'unavailable',
        reason: 'No E2E framework (playwright/cypress) detected',
        suggestion: 'Install: npm install -D @playwright/test'
      };
    }
  }

  // Calculate overall metrics
  results.passRate = this.calculateOverallPassRate(results.levelResults);
  results.coverage = await this.getCoverageIfAvailable(projectDir);
  results.defects = this.extractDefects(results.levelResults);

  return {
    content: [{
      type: 'text',
      text: JSON.stringify({
        status: 'complete',
        feature,
        passRate: results.passRate,
        coverage: results.coverage,
        defects: results.defects,
        levelResults: results.levelResults,
        report: this.generateQaReport(feature, results)
      }, null, 2)
    }]
  };
}
```

**Output Schema**:
```json
{
  "status": "complete",
  "feature": "user-auth",
  "passRate": 87.5,
  "coverage": 72.3,
  "defects": [
    { "test": "login.test.js", "error": "timeout", "severity": "high" }
  ],
  "levelResults": {
    "L1": { "total": 24, "passed": 22, "failed": 2, "passRate": 91.7 },
    "L2": { "tests": [...], "passRate": 80.0 },
    "L3_L5": { "status": "unavailable", "reason": "..." }
  },
  "report": "## QA Report\n..."
}
```

**CC와의 차이점**:
- CC: Chrome MCP를 통한 실제 브라우저 자동화 (L3-L5 완전 자동)

- CC의 Chrome MCP가 제공하는 실시간 DOM 조작/스크린샷은 불가하지만,

### 3.3 bkit_iterate — Gap Analysis + Fix + Recheck Loop

**CC 동등 기능**: pdca-iterator agent가 gap 분석 → 수정 → 재검증을 자동 반복

```javascript
async handleBkitIterate(args) {
  const { feature, action = 'analyze', maxIterations = 5, targetMatchRate = 90 } = args;
  const projectDir = process.cwd();
  
  // State file for tracking iterations
  const stateFile = path.join(projectDir, '.bkit', 'iterate', `${feature}.json`);
  let state = this.loadIterateState(stateFile);

  if (action === 'status') {
    return this.formatIterateStatus(state);
  }

  // Read design document
  const designPath = path.join(projectDir, 'docs', '02-design', 'features', `${feature}.design.md`);
  const designDoc = this.readFileIfExists(designPath);
  if (!designDoc) {
    return this.errorResult(`Design document not found: ${designPath}`);
  }

  // Extract design specifications
  const designSpecs = this.extractDesignSpecs(designDoc);
  
  // Scan implementation files
  const implFiles = this.scanImplementation(projectDir, feature);
  
  // Gap analysis: compare design specs vs implementation
  const gaps = this.analyzeGaps(designSpecs, implFiles);
  const matchRate = ((designSpecs.length - gaps.length) / designSpecs.length) * 100;

  // Update state
  state.iterations = state.iterations || [];
  state.iterations.push({
    iteration: state.iterations.length + 1,
    timestamp: new Date().toISOString(),
    matchRate,
    gapCount: gaps.length,
    totalSpecs: designSpecs.length,
    gaps: gaps.map(g => ({ spec: g.spec, status: g.status, file: g.file }))
  });
  state.currentMatchRate = matchRate;
  state.status = matchRate >= targetMatchRate ? 'pass' : 
                 state.iterations.length >= maxIterations ? 'max_iterations_reached' : 'needs_fix';

  // Save state
  fs.mkdirSync(path.dirname(stateFile), { recursive: true });
  fs.writeFileSync(stateFile, JSON.stringify(state, null, 2));

  return {
    content: [{
      type: 'text',
      text: JSON.stringify({
        matchRate: Math.round(matchRate * 10) / 10,
        iteration: state.iterations.length,
        maxIterations,
        targetMatchRate,
        status: state.status,
        gaps: gaps.slice(0, 20), // Top 20 gaps
        totalGaps: gaps.length,
        totalSpecs: designSpecs.length,
        recommendation: state.status === 'pass' 
          ? 'All gaps resolved. Proceed to QA phase.'
          : `Fix ${gaps.length} gaps and call bkit_iterate with action: "recheck"`
      }, null, 2)
    }]
  };
}
```

```javascript
async handleBkitTeamRun(args) {
  const { feature, team, task, agents: customAgents, strategy = 'leader' } = args;
  
  // Predefined team compositions (from lib/team/coordinator.js)
  const TEAMS = {
    pm: ['pm-discovery', 'pm-strategy', 'pm-research', 'pm-prd'],
    qa: ['qa-monitor', 'qa-strategist', 'code-analyzer'],
    cto: ['cto-lead', 'frontend-architect', 'security-architect', 'infra-architect'],
    custom: customAgents || []
  };

  const teamAgents = TEAMS[team] || TEAMS.custom;
  const results = [];

  if (strategy === 'leader') {
    // Leader pattern: first agent coordinates, others execute
    const leader = teamAgents[0];
    const members = teamAgents.slice(1);
    
    // Leader analyzes task
    const leaderResult = await this.runAgentInternal(leader,
      `As team leader, analyze task: "${task}" for feature "${feature}". ` +
      `You have team members: ${members.join(', ')}. ` +
      `Provide analysis and delegate sub-tasks.`,
      { feature });
    results.push({ agent: leader, role: 'leader', result: leaderResult });

    // Members execute sequentially
    for (const member of members) {
      const memberResult = await this.runAgentInternal(member,
        `Execute sub-task from leader: ${leaderResult.delegation?.[member] || task}`,
        { feature, leaderAnalysis: leaderResult });
      results.push({ agent: member, role: 'member', result: memberResult });
    }
  } else if (strategy === 'pipeline') {
    // Pipeline: each agent's output feeds into next
    let previousOutput = null;
    for (const agent of teamAgents) {
      const agentResult = await this.runAgentInternal(agent,
        previousOutput ? `Continue pipeline for "${task}". Previous stage output: ${previousOutput}` : task,
        { feature, previousOutput });
      results.push({ agent, role: 'pipeline-stage', result: agentResult });
      previousOutput = agentResult.output;
    }
  } else if (strategy === 'review') {
    // Review: all agents review same artifact independently
    for (const agent of teamAgents) {
      const reviewResult = await this.runAgentInternal(agent,
        `Review: "${task}" for feature "${feature}"`,
        { feature });
      results.push({ agent, role: 'reviewer', result: reviewResult });
    }
  }

  return {
    content: [{
      type: 'text',
      text: JSON.stringify({
        status: 'complete',
        team,
        strategy,
        agentsExecuted: teamAgents,
        results: results.map(r => ({
          agent: r.agent,
          role: r.role,
          summary: r.result?.summary || 'completed'
        })),
        summary: this.synthesizeTeamResults(results)
      }, null, 2)
    }]
  };
}
```

**CC와의 차이점**:
- CC: `Agent Teams` API로 5명 병렬 실행 + TeammateIdle 이벤트로 동적 작업 분배
- Gemini MCP: Leader pattern으로 순차 실행. 병렬 없음.
- **UX Parity: 70%** — 순차 실행으로 인한 시간 증가. 그러나 최종 산출물은 동일.

```javascript
async handleBkitAuditQuery(args) {
  const { filter = {}, limit = 50 } = args;
  const projectDir = process.cwd();
  const auditDir = path.join(projectDir, '.gemini', 'audit');
  
  if (!fs.existsSync(auditDir)) {
    return this.formatResult({ entries: [], summary: 'No audit logs found' });
  }

  // Read all JSONL files in date range
  const files = fs.readdirSync(auditDir)
    .filter(f => f.endsWith('.jsonl'))
    .sort();
  
  let entries = [];
  for (const file of files) {
    const lines = fs.readFileSync(path.join(auditDir, file), 'utf-8')
      .split('\n')
      .filter(Boolean)
      .map(line => { try { return JSON.parse(line); } catch { return null; } })
      .filter(Boolean);
    entries.push(...lines);
  }

  // Apply filters
  if (filter.agent) entries = entries.filter(e => e.agent === filter.agent);
  if (filter.phase) entries = entries.filter(e => e.phase === filter.phase);
  if (filter.tool) entries = entries.filter(e => e.tool === filter.tool);
  if (filter.keyword) {
    const kw = filter.keyword.toLowerCase();
    entries = entries.filter(e => JSON.stringify(e).toLowerCase().includes(kw));
  }
  if (filter.dateRange?.from) {
    entries = entries.filter(e => e.t >= filter.dateRange.from);
  }
  if (filter.dateRange?.to) {
    entries = entries.filter(e => e.t <= filter.dateRange.to);
  }

  // Limit and summarize
  const total = entries.length;
  entries = entries.slice(-limit); // Most recent N entries

  return {
    content: [{
      type: 'text',
      text: JSON.stringify({
        entries,
        total,
        returned: entries.length,
        summary: `Found ${total} audit entries` + 
          (filter.agent ? ` for agent "${filter.agent}"` : '') +
          (filter.phase ? ` in phase "${filter.phase}"` : '')
      }, null, 2)
    }]
  };
}
```

**UX Parity**: 95% — 기록 형식과 검색 기능 모두 CC와 동등.

```javascript
async handleBkitCheckpoint(args) {
  const { action, feature, id, label } = args;
  const projectDir = process.cwd();
  const cpDir = path.join(projectDir, '.bkit', 'checkpoints', feature);
  
  switch (action) {
    case 'create': {
      fs.mkdirSync(cpDir, { recursive: true });
      const cpId = `cp-${Date.now()}`;
      const snapshot = {
        id: cpId,
        feature,
        label: label || `Checkpoint at ${new Date().toISOString()}`,
        timestamp: new Date().toISOString(),
        pdcaState: this.readFileIfExists(
          path.join(projectDir, '.bkit', 'state', 'memory.json')),
        files: this.snapshotFeatureFiles(projectDir, feature)
      };
      fs.writeFileSync(path.join(cpDir, `${cpId}.json`), JSON.stringify(snapshot, null, 2));
      return this.formatResult({ 
        created: cpId, label: snapshot.label, 
        filesSnapshotted: snapshot.files.length 
      });
    }
    
    case 'list': {
      if (!fs.existsSync(cpDir)) return this.formatResult({ checkpoints: [] });
      const checkpoints = fs.readdirSync(cpDir)
        .filter(f => f.endsWith('.json'))
        .map(f => {
          const cp = JSON.parse(fs.readFileSync(path.join(cpDir, f), 'utf-8'));
          return { id: cp.id, label: cp.label, timestamp: cp.timestamp };
        })
        .sort((a, b) => b.timestamp.localeCompare(a.timestamp));
      return this.formatResult({ checkpoints });
    }
    
    case 'restore': {
      if (!id) return this.errorResult('Checkpoint ID required for restore');
      const cpFile = path.join(cpDir, `${id}.json`);
      if (!fs.existsSync(cpFile)) return this.errorResult(`Checkpoint not found: ${id}`);
      
      const snapshot = JSON.parse(fs.readFileSync(cpFile, 'utf-8'));
      // Restore PDCA state
      if (snapshot.pdcaState) {
        const stateDir = path.join(projectDir, '.bkit', 'state');
        fs.mkdirSync(stateDir, { recursive: true });
        fs.writeFileSync(path.join(stateDir, 'memory.json'), snapshot.pdcaState);
      }
      // Restore feature files
      for (const file of snapshot.files) {
        fs.mkdirSync(path.dirname(file.path), { recursive: true });
        fs.writeFileSync(file.path, file.content);
      }
      return this.formatResult({ restored: true, id, filesRestored: snapshot.files.length });
    }
    
    case 'detail': {
      if (!id) return this.errorResult('Checkpoint ID required');
      const cpFile = path.join(cpDir, `${id}.json`);
      if (!fs.existsSync(cpFile)) return this.errorResult(`Checkpoint not found: ${id}`);
      const snapshot = JSON.parse(fs.readFileSync(cpFile, 'utf-8'));
      return this.formatResult({
        id: snapshot.id, label: snapshot.label, timestamp: snapshot.timestamp,
        files: snapshot.files.map(f => f.path)
      });
    }
  }
}
```

**UX Parity**: 95% — CC의 state management와 기능적으로 동등.
MCP tool call만으로 checkpoint 생성/복원이 가능하므로 자동화 수준도 유사.

| **Automation Level L3-L4** | 부분 가능 | **YAGNI** | Policy deny만 가능. allow 없이 L3-L4 자동화는 사용자 혼란만 초래 |
| **실시간 메트릭 대시보드** | 가능 | **YAGNI** | 사용자가 지속 모니터링할 UI가 없음. bkit_pdca_status tool로 충분 |

## 7. Skill Enhancement Plan

### 7.1 PDCA Skill 고도화 (MCP Tool 연동)

`skills/pdca/skill.md`에 MCP tool 호출 가이드를 추가한다.
Skill 프롬프트는 코드를 실행할 수 없지만, agent에게 **"이 시점에 이 MCP tool을 호출하라"**고 지시할 수 있다.

**추가할 지시 항목**:

```markdown
### plan (Plan Phase) — MCP Enhanced
1. Verify feature name provided
2. **[MCP] bkit_checkpoint**: action="create", label="pre-plan" — 계획 시작 전 체크포인트
3. Read existing project context (package.json, GEMINI.md)
4. **Context Anchor 생성**: WHY/WHO/RISK/SUCCESS/SCOPE 테이블을 Plan 문서 상단에 배치
5. Generate Plan document
6. **[MCP] bkit_pdca_status**: action="update", phase="plan" — 상태 업데이트
7. **Checkpoint**: "이 계획으로 진행해도 되겠습니까?" — 사용자 확인 대기

### design (Design Phase) — MCP Enhanced
1. Verify Plan document exists (required)
2. **Full Upstream Context Loading**: Read Plan → Context Anchor 추출
3. **Context Anchor Embed**: Plan의 Context Anchor를 Design 상단에 복사
4. Present 3 architecture options + trade-off 분석
5. Wait for user selection
6. **Decision Record**: 선택 근거를 Design 문서에 기록
7. Generate Design document with Session Guide + Module Map
8. **[MCP] bkit_pdca_status**: action="update", phase="design"

### do (Do Phase) — MCP Enhanced
1. Verify Design document exists (required)
2. **Full Upstream Context Loading**:
   - Read PRD (docs/00-pm/{feature}.prd.md) → WHY context
   - Read Plan (docs/01-plan/features/{feature}.plan.md) → Context Anchor, Success Criteria
   - Read Design (docs/02-design/features/{feature}.design.md) → architecture decisions
3. **Decision Record Chain Display**:
   ```
   Decision Record Chain
   [PRD] Target: {segment} — {rationale}
   [Plan] Architecture: {selected option} — {rationale}
   [Design] State Mgmt: {approach} — {rationale}
   ```
4. **Context Anchor Display** from Design header
5. **Parse --scope parameter**: --scope 있으면 해당 모듈만 필터링
6. **[MCP] bkit_checkpoint**: action="create", label="pre-implementation"
7. **Checkpoint**: "이 범위로 구현을 시작해도 되겠습니까?" — 사용자 확인 대기
8. Provide implementation guide
9. **[MCP] bkit_pdca_status**: action="update", phase="do"

### check (Check Phase) — MCP Enhanced
1. **[MCP] bkit_iterate**: action="analyze" — gap 분석 실행
2. Display gap analysis results (matchRate, gaps)
3. If matchRate < targetMatchRate:
   a. Fix identified gaps
   b. **[MCP] bkit_iterate**: action="recheck"
   c. Repeat until pass or maxIterations
4. **[MCP] bkit_qa_run**: levels=[1,2] — 자동 테스트 실행
5. Display QA results
6. **[MCP] bkit_pdca_status**: action="update", phase="check"

### iterate (Iterate Phase) — MCP Enhanced
1. **[MCP] bkit_iterate**: action="status" — 현재 반복 상태 확인
2. If status != "pass":
   a. Review remaining gaps
   b. Fix gaps
   c. **[MCP] bkit_iterate**: action="recheck"
3. **[MCP] bkit_qa_run**: levels=[1,2,3] — 확장 테스트 (E2E 포함 if available)
4. **[MCP] bkit_pdca_status**: action="update", phase="iterate"

### report (Report Phase) — MCP Enhanced
1. **Full Upstream Context Loading**: PRD → Plan → Design → Analysis 전체 읽기
2. **[MCP] bkit_audit_query**: 해당 feature의 전체 audit 로그 수집
3. Generate comprehensive report (matchRate, QA results, decisions, timeline)
4. **[MCP] bkit_checkpoint**: action="create", label="final-report"
5. **[MCP] bkit_pdca_status**: action="update", phase="report"
```

### 7.2 빈 스킬 8개 SKILL.md 작성

| Skill | MCP Tool 연동 | 내용 |
|-------|-------------|------|
| `audit` | `bkit_audit_query` | audit 로그 검색/필터링 가이드. MCP tool 호출 지시 |
| `control` | `bkit_pdca_status` | L0-L2 자동화 레벨 표시/변경 (L3-L4 불가 안내) |
| `deploy` | 없음 | 배포 체크리스트 + 환경별 가이드 (자동 배포 불가) |
| `btw` | 없음 | .bkit/btw.json에 개선 제안 기록/조회 |
| `qa-phase` | `bkit_qa_run` | QA 실행 가이드. bkit_qa_run MCP tool 호출 지시 |

> 이전에는 "가이드만 제공"했던 것이 "실제 실행"으로 바뀐다.
> 예: `/audit search --agent pm-lead` → Skill이 `bkit_audit_query` MCP tool 호출 → 실제 로그 반환

### D7 상세 근거: WHY MCP-Centric

```
이전 접근 (Option C Pragmatic Hybrid):
- Skill 프롬프트가 "이렇게 하라"고 지시
- Agent가 지시를 따르길 기대 (LLM 의존)
- 자동화 = 없음 (프롬프트 가이드 = 수동)
- Hook = 관찰 + 기록 (능동적 실행 불가)

MCP-Centric 접근:
- Skill 프롬프트가 "이 MCP tool을 호출하라"고 지시
- MCP tool이 실제 실행 (Node.js = 결정론적)
- 자동화 = 높음 (MCP = 코드 실행 엔진)
- Hook = 관찰 + 기록 (변화 없음, 여전히 최소)

핵심 차이: "LLM이 알아서 하길 기대" vs "MCP tool이 확실히 실행"
```

## 13. Helper Utilities for MCP Tools

MCP tool handler에서 공통으로 사용하는 유틸리티 메서드:

```javascript
// spawn-agent-server.js SpawnAgentServer 클래스에 추가

/**
 * Read file content, return null if not found
 */
readFileIfExists(filePath) {
  try { return fs.readFileSync(filePath, 'utf-8'); }
  catch { return null; }
}

/**
 * Execute shell command and return stdout/stderr
 */
async execCommand(cmd, opts = {}) {
  const { cwd = process.cwd(), timeout = 60000 } = opts;
  return new Promise((resolve) => {
    const child = spawn('sh', ['-c', cmd], { cwd, timeout });
    let stdout = '', stderr = '';
    child.stdout.on('data', d => stdout += d);
    child.stderr.on('data', d => stderr += d);
    child.on('close', code => resolve({ stdout, stderr, code }));
    child.on('error', err => resolve({ stdout, stderr, code: -1, error: err.message }));
  });
}

/**
 * Detect test framework in project
 * @returns {'jest'|'vitest'|'node-test'|null}
 */
detectTestFramework(projectDir) {
  const pkg = this.readFileIfExists(path.join(projectDir, 'package.json'));
  if (!pkg) return null;
  const parsed = JSON.parse(pkg);
  const deps = { ...parsed.dependencies, ...parsed.devDependencies };
  if (deps.jest || deps['@jest/core']) return 'jest';
  if (deps.vitest) return 'vitest';
  // Check for Node.js built-in test runner
  if (parsed.scripts?.test?.includes('node --test')) return 'node-test';
  return null;
}

/**
 * Detect E2E framework
 * @returns {'playwright'|'cypress'|null}
 */
detectE2eFramework(projectDir) {
  const pkg = this.readFileIfExists(path.join(projectDir, 'package.json'));
  if (!pkg) return null;
  const parsed = JSON.parse(pkg);
  const deps = { ...parsed.dependencies, ...parsed.devDependencies };
  if (deps['@playwright/test']) return 'playwright';
  if (deps.cypress) return 'cypress';
  return null;
}

/**
 * Format MCP tool result
 */
formatResult(data) {
  return {
    content: [{
      type: 'text',
      text: JSON.stringify(data, null, 2)
    }]
  };
}

/**
 * Format error result
 */
errorResult(message) {
  return {
    content: [{
      type: 'text',
      text: JSON.stringify({ error: true, message }, null, 2)
    }],
    isError: true
  };
}

/**
 * Snapshot feature-related files for checkpoint
 */
snapshotFeatureFiles(projectDir, feature) {
  const files = [];
  const dirs = [
    path.join(projectDir, 'docs', '00-pm'),
    path.join(projectDir, 'docs', '01-plan', 'features'),
    path.join(projectDir, 'docs', '02-design', 'features'),
    path.join(projectDir, 'docs', '03-analysis'),
    path.join(projectDir, 'docs', '04-report')
  ];
  
  for (const dir of dirs) {
    if (!fs.existsSync(dir)) continue;
    const matches = fs.readdirSync(dir).filter(f => f.includes(feature));
    for (const file of matches) {
      const filePath = path.join(dir, file);
      files.push({ path: filePath, content: fs.readFileSync(filePath, 'utf-8') });
    }
  }
  return files;
}
```
