## 1. Intent Discovery

### 1.1 표면적 목표

"bkit이 느리다. 성능을 최적화하자."

사용자가 체감하는 문제:
- Gemini CLI에서 bkit 사용 시 첫 응답까지 시간이 오래 걸림
- 긴 대화에서 context window 압축이 빈번하게 발생
- 불필요한 context가 매 턴 반복 주입되어 토큰 낭비

### 1.2 실제 목표 (Deep Intent)

표면적 목표 뒤에 숨겨진 진짜 목표는 4가지:

**목표 A: 응답 속도 향상**
- 매 턴 고정 비용 8,613~10,113 tokens을 최소화
- Context window 150K 중 6.7%를 고정 오버헤드로 소모하는 구조 개선
- 대화 히스토리 compression 빈도 감소 → 맥락 유지력 향상

**목표 B: Dead code 정리로 유지보수성 향상**
- 17개 dead lib 모듈이 코드베이스에 잔존
- 5개 orphan hook scripts가 hooks.json에 미등록
- 5개 PM agents가 bkit-server.js AGENTS registry에 미등록
- 코드를 읽는 개발자(또는 AI)가 혼란을 겪는 구조

**목표 C: 할루시네이션/추측 방지 기능 유지** [CRITICAL]
- bkit의 핵심 가치 중 하나: "AI는 완벽하지 않다" 원칙
- core-rules.md의 "Always verify important decisions with user" 규칙
- PDCA 워크플로우 강제로 AI의 자의적 판단 방지
- **context 최적화 과정에서 이 규칙들이 제거되면 안 됨**

**목표 D: bkit 모든 기능 사용에 지장이 생기면 안 됨** [CRITICAL]
- 43개 skills, 21개 agents 모두 정상 동작 보장
- PDCA cycle 전체 플로우 유지
- Hook system (SessionStart, BeforeAgent, BeforeModel, AfterModel 등) 정상 동작
- MCP Server 도구 호출 정상 동작

### 1.3 성능 저하 원인 분석

Gemini CLI Extension의 context loading 메커니즘을 검증한 결과,
성능 저하의 근본 원인은 5가지로 분류된다:

#### 원인 1: 매 턴 고정 토큰 비용 (8,613+ tokens/turn)

Gemini CLI는 다음을 **매 API call마다** system prompt에 포함:

| 항목 | 로딩 방식 | Tokens/Turn | 비고 |
|------|----------|------------|------|
| GEMINI.md 본문 | EAGER (매 턴) | ~212 | 28줄 |
| @import commands.md | EAGER (매 턴) | ~566 | GEMINI.md에 인라인 |
| @import core-rules.md | EAGER (매 턴) | ~385 | GEMINI.md에 인라인 |
| Skills metadata (43개) | EAGER (매 턴) | ~5,700 | 이름+설명만, ~130 tokens/skill |
| Agents metadata (21개) | EAGER (매 턴) | ~1,750 | ~83 tokens/agent |
| **소계 (고정)** | | **~8,613** | **150K의 5.7%** |

**핵심 발견**: GEMINI.md의 @import는 파일 내용을 인라인하여 **매 API call에 포함**된다.
Skills는 name + description만 로드하지만, 43개 skill의 메타데이터 합계가 5,700 tokens로
고정 비용의 66%를 차지한다.

#### 원인 2: SessionStart Hook Context 잔존

```
SessionStart additionalContext → 대화 히스토리에 주입 → compression까지 잔존
```

현재 `session-start.js`의 `generateDynamicContext()`가 주입하는 context:
- `buildCoreRules()`: ~300 tokens — **core-rules.md @import와 완전 중복**
- `buildOnboardingSection()`: ~200 tokens — 첫 턴에만 필요
- `buildOutputStyleSection()`: ~150 tokens — 스타일 규칙
- `buildAvailableSkillsSection()`: ~200 tokens — level별 skill 목록
- `buildPdcaStatusSection()`: ~100 tokens — 현재 상태
- `loadPhaseAwareContext()`: ~500 tokens — phase별 context 파일들
- Tracker context: ~100 tokens

**총 SessionStart 주입량**: ~1,500 tokens

이 중 `buildCoreRules()` ~300 tokens는 @import된 core-rules.md와 **내용이 동일**하여
명백한 중복 낭비이다.

#### 원인 4: 3중 중복 PDCA Rules

PDCA 규칙이 3곳에 분산 존재:

| 위치 | 로딩 방식 | 비용 | 내용 |
|------|----------|------|------|
| `.gemini/context/core-rules.md` | @import (매 턴 EAGER) | ~385 tokens/turn | PDCA Core Rules + Feature Usage Report + Executive Summary |
| `.gemini/context/pdca-rules.md` | Hook conditional (phase별) | ~200 tokens (조건부) | PDCA 상세 규칙 (32줄) |
| `session-start.js buildCoreRules()` | SessionStart context | ~300 tokens (세션당) | core-rules.md 내용 중복 재구성 |

**중복 낭비**: 최소 ~300 tokens (buildCoreRules) + 조건부 ~200 tokens (pdca-rules.md)
→ core-rules.md 하나로 통합하면 ~500 tokens 절감 가능

#### 원인 5: 미사용 Context 파일

| 파일 | Tokens | 로딩 방식 | 실제 사용 여부 |
|------|--------|----------|--------------|
| agent-triggers.md | ~788 | Hook conditional (idle phase) | 코드에서 미사용 — intent matching은 lib/intent로 동적 처리 |
| skill-triggers.md | ~912 | Hook conditional (do phase) | 코드에서 미사용 — intent matching은 lib/intent로 동적 처리 |

**합계**: ~1,700 tokens가 코드 로직과 무관하게 context에 주입될 수 있음

## 2. 현재 상태 심층 분석

### 2.1 Gemini CLI Context Loading 메커니즘 (검증 완료)

Gemini CLI Extension의 context loading은 4개 계층으로 구성된다:

```
┌─────────────────────────────────────────────────────┐
│ Layer 1: GEMINI.md + @import                        │
│ ══════════════════════════════════════════           │
│ - 재귀적 탐색 후 concatenation (session start)       │
│ - @import = 파일 내용 인라인                         │
│ - 매 API call system prompt에 포함 (EAGER)           │
│ - bkit 현재: commands.md(566) + core-rules.md(385)  │
│ - Total: ~1,163 tokens/turn                         │
│                                                     │
│ [결론] @import는 "매 턴 비용"이므로 최소화 필수      │
├─────────────────────────────────────────────────────┤
│ Layer 2: Skills (Progressive Disclosure)             │
│ ══════════════════════════════════════════           │
│ - name + description만 system prompt (~130/skill)   │
│ - SKILL.md body는 activate_skill 시에만 로드         │
│ - 43 skills × ~130 = ~5,700 tokens/turn             │
│                                                     │
│ [결론] skill 수 자체가 고정 비용 결정                │
│ → 미사용 skill 비활성화가 가장 효과적               │
├─────────────────────────────────────────────────────┤
│ Layer 3: Agents (Metadata in System Prompt)          │
│ ══════════════════════════════════════════           │
│ - ~50-100 tokens per agent description              │
│ - 21 agents × ~83 = ~1,750 tokens/turn              │
│                                                     │
│ [결론] Agent description 압축 + 미사용 agent 제거    │
├─────────────────────────────────────────────────────┤
│ Layer 4: Hook Context Injection                      │
│ ══════════════════════════════════════════           │
│ - SessionStart: 대화 히스토리에 잔존 (compression까지)│
│ - BeforeAgent: "this turn only" (가장 효율적)        │
│ - AfterTool: 단일 tool result에 append               │
│ - Context는 턴 간 누적 안 됨                        │
│                                                     │
│ [결론] SessionStart → BeforeAgent 전환이 핵심        │
└─────────────────────────────────────────────────────┘
```

### 2.4 모듈 분류 및 처리 방안 (Dead Code 재분석)

> **중요**: 초기 감사에서 "17개 dead code"로 분류했으나, 심층 분석 결과

> 진짜 삭제 대상은 1개뿐이다.

4. **미래 대비**: Gemini CLI v0.38+에서 context budget이 더 tight해질 가능성에 대비

**Option C 채택 시 제약 조건**:
- **bkit 모든 기능 사용에 지장이 생기면 안 됨** — skill 통합은 하되, 기능 삭제는 안 함

### 4.1 GEMINI.md @import 최적화

#### 현재 상태

```markdown
# GEMINI.md (현재 v2.0.4)
@.gemini/context/commands.md    ← 566 tokens, 매 턴 EAGER 로드
```

**문제**: Gemini CLI 공식 문서 검증 결과, @import는 파일 내용을 인라인하여
매 API call의 system prompt에 포함한다. 즉 **매 턴 951 tokens 고정 비용**.

#### 분석: 정말 매 턴 필요한가?

**commands.md (566 tokens)**:
- 사용자가 "어떤 명령어가 있어?" 물어볼 때만 필요
- 대부분의 턴에서는 사용자가 이미 명령어를 알고 있음
- 빈도: 세션당 1-2회 (전체 턴의 ~5%)
- **결론**: 매 턴 로드는 낭비. Skill activation 시 로드로 전환

**core-rules.md (385 tokens)**:
- PDCA 워크플로우 규칙 + Feature Usage Report 포맷 + Executive Summary 규칙
- **Feature Usage Report는 매 턴 필요** (응답 끝에 항상 포함)
- PDCA 규칙은 PDCA 작업 시에만 필요
- **결론**: Feature Usage Report 규칙만 GEMINI.md 본문으로 승격, 나머지는 조건부 로드

#### 개선안: GEMINI.md 슬림화

```markdown

```
bkit Feature Usage

```

---

*bkit Vibecoding Kit v2.1.0 - Gemini CLI Native Edition*
*Copyright 2024-2026 POPUP STUDIO PTE. LTD.*
```

- @import core-rules.md 제거 → 핵심 5줄은 GEMINI.md 본문으로 승격
- Feature Usage Report 포맷은 GEMINI.md에 직접 포함 (매 턴 필요하므로)

**토큰 변화**:
- Before: 212 (본문) + 566 (commands) + 385 (core-rules) = **1,163 tokens/turn**

### 4.4 SessionStart Hook Context 최소화

#### 현재 문제

`session-start.js`의 `generateDynamicContext()`가 **SessionStart** hook으로 주입하는 context는
대화 히스토리에 추가되어 **compression이 발생할 때까지 잔존**한다.

Gemini CLI 검증 결과:
- SessionStart additionalContext → 대화 히스토리에 영구 잔존 (compression까지)

- Context는 턴 간 누적되지 않음

#### 검증: 왜 미사용인가?

`before-agent.js` (BeforeAgent hook)에서 intent detection을 수행할 때,
`lib/intent/trigger.js`가 사용자 입력을 분석하여 적절한 skill/agent를 매칭한다.

`.gemini/context/agent-triggers.md`나 `skill-triggers.md`를 **파일로 읽지 않는다**.

1. Phase-Aware Context로 모델에 주입되어 "참고 정보"로만 사용
2. 실제 trigger 매칭 로직과 sync되지 않을 수 있음 (문서-코드 불일치 위험)

### 8.1 포팅하면 안 되는 것 (bkit-claude-code에서)
