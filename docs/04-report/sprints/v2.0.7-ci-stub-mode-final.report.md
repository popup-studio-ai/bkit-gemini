## 3. Architecture

### 3.1 Module boundary

```
Scenario test
   ↓ recordOrReplay({args, env, cwd, label})
lib/test/llm-replay.js
   ├─ computeRequestHash() — SHA256(allowlisted env, sorted)
   ├─ getTestMode() — BKIT_TEST_MODE: stub|record|live (default live)
   ├─ stub  → loadFixture() → return {fromFixture: true}
   ├─ record → _runLiveImpl() → saveFixture() → return
   └─ live  → _runLiveImpl() → return {fromFixture: false}
```

### 3.2 Hash 안정성 보증 (LR-01~04 동적 강제)

- args 순서 보존 (LR-02)
- env allowlist만 (LR-03) — PATH/HOME/SECRET 등은 hash 무영향
- cwd basename만 (LR-04) — absolute path 누설 방지

### 3.3 Fail-fast 원칙 (LR-05)

stub mode + fixture miss → `Error` (code: `BKIT_STUB_MISS`) + hint 메시지. silent fallback to live 금지.

### 3.4 Test injection (`_setRunLiveImpl`)

`spawn('gemini', ...)` 실 호출을 우회하는 injection point. 동적 회귀 test는 spawn 0회로 모드별 동작 검증 (LR-06, 07, 08).

## 7. Lessons Learned

1. **Hash 입력 normalize의 중요성** — env allowlist + cwd basename 정책으로 false-miss(개발자 환경 차이) 방지. fixture는 cross-OS, cross-developer로 재현 가능.
2. **Fail-fast > silent fallback** — stub mode에서 fixture miss 시 즉시 throw + hint. CI에서 누락이 silent하게 가려지면 회귀 검증의 의미가 사라짐.
3. **Test injection pattern** — `_setRunLiveImpl(fn)`으로 동적 회귀 test가 spawn 0회로 record/replay 분기 검증. 실 LLM 비용 없이 framework 자체 동작 보장.
4. **Sprint scope 분할** — framework 입증과 시나리오 자동화는 분리. 한 sprint에 모두 묶으면 캡처/검증/yaml 작성으로 단일-day 범위 초과 + 산만한 incremental 진행 위험.
