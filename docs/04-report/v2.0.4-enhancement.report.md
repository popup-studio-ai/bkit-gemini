## 1. Executive Summary

### 1.1 Overview

bkit-gemini v2.0.4는 Gemini CLI Extension의 구조적 한계를 MCP (Model Context Protocol) Server로
돌파한 아키텍처 전환 릴리스이다.

Gemini CLI Extension은 본질적으로 **관찰자 + 조언자** 역할에 한정된다.
Hook은 `deny` only이며, Skill은 프롬프트 텍스트일 뿐 코드 실행 능력이 없다.
반면 bkit-claude-code는 Plugin 시스템을 통해 `Task()`, `AskUserQuestion()`,
`PostToolUse` Hook 등 런타임 제어가 가능하여 체계적인 PDCA 워크플로우 UX를 제공한다.

**v2.0.4의 핵심 결정**: Extension이 할 수 없는 것을 MCP Server가 대신한다.
MCP Server는 별도 Node.js 프로세스로 실행되어 `child_process`, `fs`, `http`에
자유롭게 접근할 수 있으며, 구조화된 결과를 main agent에 반환한다.

이 전략의 결과:
- **43/43 Skills** 완전 구현 (v2.0.3의 27/35에서 100%로)
- **12 MCP Tools** (기존 6 + 신규 6)
- **~4,000 LOC Dead Code 제거** (net -800 LOC)
- **bkit-claude-code 대비 ~90% UX 동등성** (v2.0.3의 ~40%에서)

## 2. Architecture Decision: MCP-Centric

### 2.1 Why MCP Server?

bkit-gemini v2.0.3까지의 가장 큰 문제는 **Extension의 구조적 한계**였다.

| Extension 제약 | 의미 | MCP Server 대안 |
|---------------|------|----------------|
| Hook은 `deny` only | 도구 실행을 차단만 가능, 허용/변경 불가 | MCP tool에서 자유로운 로직 실행 |
| Skill은 텍스트 전용 | `.md` 프롬프트, 코드 실행 능력 없음 | MCP tool이 코드 실행 후 결과 반환 |
| Hook 간 통신 불가 | `before-tool` → `after-tool` 상태 전달 경로 없음 | MCP Server 프로세스 내 state 공유 |
| 병렬 에이전트 불가 | `spawn_agent`는 순차 실행만 지원 | MCP 내부 `child_process` 병렬 실행 |
| Hook 60초 타임아웃 | 장시간 작업 불가 (테스트, 빌드 등) | MCP tool은 비동기, 시간 제한 없음 |
| stdout JSON만 반환 | 구조화된 응답 포맷 제한 | MCP tool은 rich structured result 반환 |

**결론**: 복잡한 로직은 MCP 서버에, 단순 관찰/기록은 hooks에 배치하는 것이
Gemini CLI Extension 아키텍처에서의 최적해이다.

### 2.2 Design Options 평가

v2.0.4 Design Phase에서 3가지 Option을 평가하였다:

| Option | 전략 | 공수 | 장점 | 단점 |
|--------|------|------|------|------|
| A: Extension 확장 | 기존 hooks/skills 강화 | 15h | 변경 최소 | dead code 해결 불가, UX 개선 한계 |
| B: Multi-Server | 기능별 MCP 서버 분리 | 45h | 모듈성 | 과도한 복잡성, 서버 관리 부담 |
| **C: Unified Replace** | **단일 MCP 서버로 통합** | **30-38h** | **dead code 해결 + UX 90%** | **단일 프로세스 의존** |

**Option C 선택 근거**:
1. 기존 `spawn-agent-server.js` (907줄)를 리팩토링 기반으로 활용
2. 단일 서버가 상태 공유에 유리 (audit trail, checkpoint 등)
3. dead code를 삭제하고 필요한 기능만 MCP tool로 재구현
4. `.gemini/settings.json`에 등록 한 줄로 전체 기능 활성화

### 2.3 3-Layer Architecture

v2.0.4의 아키텍처는 세 레이어로 구성된다:

```
┌──────────────────────────────────────────────────────────┐
│                    Layer 1: Hooks                        │
│                  (관찰 / 기록 / 차단)                     │
│                                                          │
│  before-agent.js   lib/intent import 기반 라우팅          │
│  after-tool.js     inline 10줄 audit trail               │
│  session-start.js  agent 활성화 + context 주입            │
│  before-tool.js    위험 명령 차단 (deny only)              │
└──────────────────────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│                    Layer 2: Skills                        │
│                  (가이드 / UX 프롬프트)                    │
│                                                          │
│  pdca/skill.md     Context Anchor, Decision Record,      │
│                    Checkpoint, --scope 가이드             │
│  43 skills         MCP tool 호출 가이드 포함               │
└──────────────────────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│                  Layer 3: MCP Server                      │
│                  (실행 / 오케스트레이션)                    │
│                                                          │
│  bkit-server.js (740 LOC)                                │
│  ├── spawn_agent, list_agents, get_info (기존)            │
│  ├── team_create, team_assign, team_status (기존)         │
│  ├── bkit_pm_run        PM pipeline 실행                  │
│  ├── bkit_qa_run        QA test suite 실행                │
│  ├── bkit_iterate       Gap analysis + 반복                │
│  ├── bkit_team_run      Team 오케스트레이션                 │
│  ├── bkit_audit_query   Audit trail 검색                   │
│  └── bkit_checkpoint    Checkpoint 생성/복원               │
│                                                          │
│  mcp/tools/                                              │
│  ├── pm-pipeline.js        (330 LOC)                     │
│  ├── qa-runner.js          (511 LOC)                     │
│  ├── gap-analyzer.js       (244 LOC)                     │
│  ├── audit-store.js        (153 LOC)                     │
│  └── checkpoint-manager.js (192 LOC)                     │
└──────────────────────────────────────────────────────────┘
```

### 2.4 Dead Code 방지 전략

**이전 시도의 실패 교훈**:
- `lib/team/`, `lib/task/` 모듈을 만들고 hooks에서 import하려 했으나,
  Extension hooks의 제약으로 실제 연결이 불가하여 전량 dead code 화
- `skill-orchestrator.js` (754줄)도 hooks에서 호출할 수 없어 dead code

**v2.0.4 해결 방식**:
- MCP 서버 전용 모듈을 `mcp/tools/`에 배치
- hooks가 import하지 않아도 MCP 서버가 직접 `require()` 사용
- hooks는 최소한의 inline 코드만 (after-tool.js 10줄 audit)
- 이 구조에서는 **모듈이 곧 MCP tool의 구현체**이므로 dead code가 될 수 없음

### 3.3 Phase 3: Integration (Skill + Hook + Agent)

Phase 3는 MCP 서버와 기존 Extension 컴포넌트를 연결하는 통합 단계.

#### 3.3.2 Hook 최소 변경

Hook 변경은 의도적으로 **최소한**으로 유지하였다:

이 접근의 의미: hooks는 "관찰만, 실행하지 않는다"는 원칙 유지.

### 6.2 Parity 분석

**100% 달성 영역 (4건)**: Context Anchor, Decision Record, Checkpoint, --scope
- CC는 Hook/Plugin API로 자동화하지만, Gemini는 Skill prompt guide로 동일한 **사용자 경험**을 제공
- 메커니즘은 다르나 결과적 UX는 동등

**80-95% 영역 (4건)**: PM, QA, Iterate, Audit
- MCP tool이 핵심 로직을 실행하나, CC의 병렬 에이전트 대비 순차 실행으로 속도 차이
- QA는 shell 기반 실행으로 CC의 에이전트 기반 대비 유연성이 약간 부족

**0% 영역 (1건)**: Agent Teams
- Gemini CLI는 `spawn_agent` 순차 실행만 지원 (병렬 에이전트 미지원)
- Google issue [#19430](https://github.com/google-gemini/gemini-cli/issues/19430)에서 논의 중
- MCP 내부 `child_process` 병렬은 가능하나, Gemini agent 컨텍스트 밖에서의 실행이므로 true parity는 아님

## 7. Lessons Learned

### 7.1 이전 버전에서 배운 것

| 실패 패턴 | v2.0.3 이하에서의 사례 | v2.0.4 대응 |
|----------|----------------------|------------|
| Lib 모듈 → Dead Code | `lib/team/` 12 files 전량 미사용 | `mcp/tools/`로 이동, MCP 서버가 직접 사용 |
| CC 기능 직접 포팅 | `skill-orchestrator.js` 754줄 | Extension 아키텍처에 맞게 재설계 |
| Hook 과부하 | hooks에 복잡한 로직 시도 | hooks는 10줄 이내 inline, 나머지는 MCP |
| Empty stub 방치 | 8 skill dirs 빈 상태 | 전수 구현 (SKILL.md + MCP tool 연동) |

### 7.2 v2.0.4에서 잘한 것

1. **Dead Code First**: 새 코드 추가 전에 ~4,000 LOC dead code를 먼저 정리.
   코드베이스가 깨끗한 상태에서 시작하여 구현 혼란 방지.

2. **MCP Server = Escape Hatch**: Extension의 한계를 "극복"하려 하지 않고,
   MCP 서버라는 정당한 우회 경로를 활용. Gemini CLI가 공식 지원하는 메커니즘이므로
   향후 CLI 업데이트에도 안전.

3. **모듈 배치 전략**: `mcp/tools/`에 배치하여 구조적으로 dead code 불가능.
   모듈은 MCP 서버의 tool handler이므로 반드시 호출됨.

4. **Hook 변경 최소화**: `after-tool.js`에 10줄 inline audit만 추가.
   Hook 코드가 복잡해질수록 Extension 업데이트 시 깨질 위험이 높으므로,
   최소한의 접점만 유지.

5. **Gap Analysis 내장**: Check Phase에서 계획 vs 구현 100% 일치를 검증.
   2건의 미삭제 파일을 발견하고 즉시 수정.

6. **QA에서 Critical 발견**: 8건 중 2건의 Critical 이슈를 QA 단계에서 발견.
   nvm PATH 이슈와 MCP notification 미처리는 QA 없이 발견하기 어려운 문제.

### 7.3 다음 버전을 위한 제안

| # | 제안 | 근거 | 우선순위 |
|---|------|------|---------|
| 1 | Legacy test 정리 | 56건 skip 테스트가 CI noise 유발 | P1 |
| 2 | Gemini CLI v0.38.0 `ContextCompressionService` 활용 | 긴 세션에서 컨텍스트 최적화 | P2 |
| 3 | A2A Protocol 원격 에이전트 위임 | 안정화 시 CC 수준의 병렬 실행 가능 | P3 |
| 4 | Agent Teams (#19430) 모니터링 | 병렬 에이전트 지원 시 UX parity 100% 가능 | P3 |
| 5 | MCP tool 실제 프로젝트 validation | pm-pipeline, qa-runner 품질 검증 | P1 |

## 11. Conclusion

bkit-gemini v2.0.4는 **MCP Server-Centric Architecture** 전환이라는 전략적 결정을 통해
Gemini CLI Extension의 구조적 한계를 돌파한 릴리스이다.

**핵심 성과**:

1. **Dead Code 제거**: ~4,000 LOC 삭제, net -800 LOC. 코드베이스가 더 작고 깨끗해짐.

2. **MCP 서버 확립**: 12 tools (6 기존 + 6 신규), 5개 전용 module.
   PM/QA/Iterate/Audit/Checkpoint 오케스트레이션이 실제로 동작.

3. **UX Parity 90%**: bkit-claude-code 대비 ~40% → ~90%로 대폭 개선.
   Context Anchor, Decision Record, Checkpoint, --scope 모두 지원.

4. **Skills 100%**: 43/43 스킬 완전 구현 (이전 27/35의 77%에서).

5. **QA 검증 완료**: 21 test suites, 870 automated tests, 8건 이슈 발견/수정.

남은 10% UX gap은 Gemini CLI 자체의 병렬 에이전트 미지원으로 인한 구조적 한계이며,
Google의 Agent Teams 기능 (#19430) 진행에 따라 해소될 수 있다.

v2.0.4는 "Gemini CLI의 제약을 인정하되, MCP 서버로 우회한다"는 실용적 전략이
Extension 기반 AI 도구 개발에서 유효함을 입증한 사례이다.
