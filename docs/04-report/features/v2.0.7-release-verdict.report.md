## 2. Critical 발견 (Hotfix 권고)

### 2.1 [CRITICAL-1] PDCA Schema Mismatch 부분 잔존

**상태**: Critical (Silent Degradation), **Fix 비용**: 30분, **Hotfix 대상**

#### 사실 (코드 검증 완료)

`lib/pdca/status.js:10,44` v3 schema 정의:
```js
activeFeatures: [],   // v3 schema — array of names
features: {},         // v3 schema — per-feature data map
```

그러나 다음 8라인이 v3 array를 v2 map처럼 사용:

| 파일 | 라인 | 코드 | 결함 |
|------|------|------|------|
| `hooks/scripts/session-start.js` | 100 | `const active = pdcaStatus.activeFeatures \|\| {};` | array에 `\|\| {}` → 빈 array 통과, 이후 `active[name]` 항상 undefined |
| `hooks/scripts/session-start.js` | 119 | `(pdcaStatus.activeFeatures \|\| {})[pdcaStatus.primaryFeature]?.phase` | array indexing → 항상 undefined |
| `hooks/scripts/session-start.js` | 250 | `const active = pdcaStatus?.activeFeatures \|\| {};` | 동일 |
| `hooks/scripts/session-start.js` | 423 | `const active = pdcaStatus.activeFeatures \|\| {};` | 동일 |
| `hooks/scripts/session-start.js` | 535 | `Object.keys(pdcaStatus.activeFeatures).length` | array에 `Object.keys()` → 0 항상 |
| `hooks/scripts/after-agent.js` | 164 | `const active = pdcaStatus.activeFeatures \|\| {};` | 동일 |
| `hooks/scripts/after-agent.js` | 210 | `const active = pdcaStatus.activeFeatures \|\| {};` | 동일 |
| `hooks/scripts/after-agent.js` | 255 | `const active = pdcaStatus.activeFeatures \|\| {};` | 동일 |

#### 영향

- `loadPdcaStatus()`가 v3 schema로 정규화하므로 read는 무사 → 사용자 visible bug 거의 0
- 그러나 **silent degradation**:
  - SessionStart header의 "Returning User" 검출 실패
  - phase tracker injection 누락
  - `after-agent.js`의 matchRate write가 빈 객체에 mutation → schema 오염 가능
- 회귀 테스트가 catch 못한 이유: 정규화된 read path와 graceful fallback에 의존

#### 권고 Fix (30분, 1-line replace × 8)

```js
// Before
const active = pdcaStatus.activeFeatures || {};
const phase = active[primary]?.phase;

// After
const phase = pdcaStatus.features?.[primary]?.phase;
```

`session-start.js:535`만 특별 처리:
```js
const activeCount = Array.isArray(pdcaStatus.activeFeatures)
  ? pdcaStatus.activeFeatures.length
  : 0;
```

### 2.2 [CRITICAL-2] BeforePhase Hook 공식 미지원 확정

**상태**: Critical (Silent No-op), **Fix 비용**: 1시간, **Hotfix 대상**

#### 사실 (공식 문서 검증)

Gemini CLI v0.42.0 공식 hook event 목록 (출처: [geminicli.com/docs/hooks/reference](https://geminicli.com/docs/hooks/reference/), [GitHub docs/hooks/reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/hooks/reference.md)):

```
1. BeforeTool             7. AfterModel
2. AfterTool              8. SessionStart
3. BeforeAgent            9. SessionEnd
4. AfterAgent            10. Notification
5. BeforeModel           11. PreCompress
6. BeforeToolSelection
```

**`BeforePhase` 부재**. v0.43.0-preview.0 changelog에도 추가 없음. 가장 가까운 요청은 issue #18493 (Pre-Plan/Post-Plan hook feature request, open 2026-02 status). bkit `hooks/hooks.json:135-146` 의 `BeforePhase` 등록은 **bkit custom 명명**이며 Gemini CLI가 silently 무시합니다.

#### bkit 코드 의존

- `hooks/scripts/before-phase.js` (308 lines): PDCA phase 전환 gate (M1/M3/M5/M7/M9 평가)
- `lib/core/audit-log.js:21`: `hook?: 'BeforePhase'` JSDoc 명세
- S7 design 약속: AC-T2 (L4 phase 자동 전환), AC-T14 (before-phase 작동)

#### 영향

- L4 자동화 모드에서 phase 자동 전환 시 quality gate 평가 **silent skip**
- 사용자 의도된 보호 (Plan→Design 진입 시 design 문서 존재 검증) 작동 안 함
- **그러나** L0~L2 모드에서는 사용자가 `/pdca` 슬래시 명령으로 수동 전환하므로 영향 0
- bkit이 L4 trust 미달인 사용자에게는 사실상 무영향, **L4 자동 모드 사용자만 silent degradation**

#### 권고 Fix (1시간)

**옵션 A (권고)**: `BeforePhase` → `BeforeAgent`로 의미 매핑
- `BeforeAgent`는 user prompt 후 planning 직전 트리거
- before-phase.js의 phase transition detection logic을 prompt parse 단계로 이동
- prompt에서 `/pdca {phase}` 또는 자연어 phase 의도 검출 시 gate 평가
- `hooks/hooks.json:135-146` 항목 제거, `BeforeAgent` 항목에 통합

**옵션 B**: `BeforePhase` → `SessionStart` 매핑
- 매 세션 시작 시 직전 phase 종료 조건 1회 평가
- pause/resume 사용자에 적합

**옵션 C (장기)**: GitHub issue #18493에 bkit 사용 사례 댓글 + PR 제출 (Pre-Plan / Post-Plan hook 도입)

**즉시 권고**: 옵션 A로 v2.0.7.1 hotfix, 옵션 C는 v2.1.0 sprint로 분리.

#### 워크어라운드

영향받는 subagent의 frontmatter `tools` 필드에 명시:

#### 권고

### 3.2 [MAJOR-2] Nested `gemini -p` write_file 거부

**상태**: Major (의도된 보안 동작), **Fix 비용**: 정책 엔진 rule 1시간, **v2.1.0 sprint**

#### 사실

- Gemini CLI non-interactive 모드에서 write_file은 기본 `ask_user` → headless 환경에서 deny

#### 영향

- bkit 자동화된 파일 수정 워크플로가 nested 실행 시 silent 실패
- 사용자가 outer agent에서 write를 위임할 경우 영향
- **그러나** 의도된 보안 정책이며 bypass는 보안 침해

#### 권고

- v2.0.7 Known Issues에 명시 (security feature로 표기)
- v2.1.0 sprint: 정책 엔진에 `subagent: "<allowlist>"` rule 추가 검토 (보안 영향 평가 필수)
- 아키텍처 권고: write 책임을 outer agent로 이전하여 nested write 회피

### 9.2 핵심 메시지

1. **v2.0.7은 외부 릴리즈 결격 사유가 없습니다.** S1~S7 7개 스프린트 누적 산출물은 v2.0.6 대비 유의미한 진전을 달성했고, 회귀 97.1% + manual 61 PASS는 충분히 견고합니다.

2. **그러나 2건의 silent failure가 사용자 신뢰도 손상 위험을 만듭니다.** 둘 다 1시간 이내 fix 가능한 mechanical change이므로, v2.0.7.1 hotfix (1일) 후 정식 배포가 비용 대비 안전합니다.

3. **나머지 5건은 공식 미정의 영역, 의도된 보안 동작, 또는 자체 KPI 미달**로 Known Issues 문서화로 충분합니다. v0.43.0 stable에서도 해결되지 않을 항목이 다수이므로 대기는 비효율적입니다.

4. **컨텍스트 엔지니어링 면에서 bkit은 Gemini CLI 공식 "Conductor" 패러다임과 정렬**되어 있으며, Progressive Disclosure / Hierarchical Memory / Agent Skills 모두 v0.42.0과 호환됩니다.
