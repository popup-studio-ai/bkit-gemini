## 3. P0 Critical Tests -- Session & Hook System

> Session lifecycle와 Hook 파이프라인은 bkit의 핵심 인프라.
> 이 테스트가 실패하면 나머지 모든 기능이 동작하지 않음.

### TC-P0-01: Extension Loading

Extension이 Gemini CLI에 정상 로드되는지 확인.

```bash
gemini -l 2>&1 | grep -i bkit
```

- **기대 결과**: "bkit" 이름이 extension 목록에 표시
- **통과 기준**: Extension 이름 확인 가능
- **실패 시 조치**: `~/.gemini/extensions/bkit` symlink 확인, `gemini-extension.json` 문법 검증

### TC-P0-02: Session Initialization (session-start.js)

Session 시작 시 `session-start.js` hook이 정상 실행되는지 확인.

```bash
gemini -p "What is your current working directory? Just tell me the path." --yolo -o text 2>&1
```

- **기대 결과**: Session-start hook 실행, stderr에 에러 없음, 응답 수신
- **통과 기준**: Working directory 경로가 정상 응답됨
- **검증 포인트**: session-start.js가 phase-aware context를 주입했는지 확인

### TC-P0-03: PDCA Status 조회

PDCA 상태 조회 기능이 정상 동작하는지 확인.

```bash
gemini -p "Run /pdca status and show me the current PDCA status." --yolo -o text 2>&1
```

- **기대 결과**: PDCA status 정보 표시 (phase, feature 등)
- **통과 기준**: Status 정보가 화면에 출력됨
- **검증 포인트**: `.pdca-status.json` 파일 기반 상태 반영 여부

### TC-P0-04: Dangerous Command Blocking (before-tool.js)

위험한 shell command가 `before-tool.js`에 의해 차단되는지 확인.

```bash
gemini -p "Run this shell command: rm -rf /" --yolo -o text 2>&1
```

- **기대 결과**: `before-tool.js`가 명령을 차단, 경고 메시지 표시
- **통과 기준**: 명령이 실행되지 않고, 차단/경고 메시지 출력
- **검증 포인트**: `bkit.config.json`의 `permissions.run_shell_command(rm -rf*)`: `deny` 정책 적용

### TC-P0-05: MCP Server Tools 등록 확인

MCP server가 등록한 도구들이 사용 가능한지 확인.

```bash
gemini -p "List all available MCP tools. Just show tool names." --yolo -o text 2>&1
```

- **기대 결과**: bkit_audit_search, bkit_checkpoint_list, bkit_code_quality, bkit_gap_analysis, bkit_regression_rules, bkit_pdca_status, bkit_feature_list 등 표시
- **통과 기준**: 최소 5개 이상의 bkit MCP tool 이름 확인
- **검증 포인트**: `mcp/bkit-server.js` 및 `mcp/tools/` 하위 모듈 정상 로드

### TC-P0-06: Audit Trail 생성 (after-tool.js)

도구 실행 후 `after-tool.js`가 audit log를 기록하는지 확인.

```bash
gemini -p "Read the file gemini-extension.json and tell me the version." --yolo -o text 2>&1
# 확인:
ls .gemini/audit/*.jsonl 2>/dev/null
```

- **기대 결과**: `after-tool.js`가 audit JSONL 항목 생성
- **통과 기준**: `.gemini/audit/` 디렉토리에 `.jsonl` 파일 존재
- **검증 포인트**: JSONL 항목에 timestamp, tool name, result 포함

### TC-P0-07: BeforeAgent Hook (before-agent.js)

Agent 호출 전 intent detection hook이 실행되는지 확인.

```bash
gemini -p "코드 리뷰를 해줘. mcp/bkit-server.js 파일을 분석해." --yolo -o text 2>&1
```

- **기대 결과**: `before-agent.js`가 intent를 감지하고 적절한 agent 추천
- **통과 기준**: 코드 분석 관련 응답이 돌아옴
- **검증 포인트**: intent detection 결과가 agent routing에 반영

### TC-P0-08: BeforeModel Hook (before-model.js)

Model 호출 전 context injection hook이 실행되는지 확인.

```bash
gemini -p "Tell me about bkit version and its main features." --yolo -o text 2>&1
```

- **기대 결과**: `before-model.js`가 context를 주입, bkit 관련 정보 포함 응답
- **통과 기준**: bkit v2.0.4 정보가 응답에 포함
- **검증 포인트**: GEMINI.md context가 model에 전달되었는지 확인

### TC-P0-09: AfterModel Hook (after-model.js)

Model 응답 후 후처리 hook이 실행되는지 확인.

```bash
gemini -p "Create a simple hello world function in JavaScript." --yolo -o text 2>&1
```

- **기대 결과**: `after-model.js`가 응답 후처리 실행
- **통과 기준**: 응답 정상 수신, Feature Usage report 포함 여부 확인
- **검증 포인트**: output style 적용 여부

### TC-P0-10: SessionEnd Hook (session-end.js)

세션 종료 시 cleanup hook이 실행되는지 확인.

```bash
gemini -p "Say goodbye." --yolo -o text 2>&1
# session-end.js는 세션 종료 시 자동 실행
# 검증: 에러 로그 없음
```

- **기대 결과**: `session-end.js` 정상 실행, 리소스 정리
- **통과 기준**: 에러 없이 세션 종료
- **검증 포인트**: agent-memory, audit log 정리 여부

## 4. P0 Critical Tests -- PDCA Core Workflow

> PDCA cycle은 bkit의 핵심 방법론. Plan -> Design -> Do -> Check -> Act -> Report 전 과정 검증.

### TC-P0-11: PDCA Plan 생성

새 feature의 Plan 문서를 생성하는 기능 검증.

```bash
gemini -p "/pdca plan test-qa-feature-01 -- Create a simple plan document for a test feature. Keep it minimal, just test that the document gets created." --yolo -o text 2>&1
# 확인:
ls docs/01-plan/features/test-qa-feature-01.plan.md 2>/dev/null
```

- **기대 결과**: Plan 문서가 `docs/01-plan/features/` 에 생성
- **통과 기준**: 파일 존재 + 내용에 feature 이름 포함
- **검증 포인트**: `plan.template.md` 템플릿 구조 반영 여부

### TC-P0-12: PDCA Design 생성

Plan 기반으로 Design 문서를 생성하는 기능 검증.

```bash
gemini -p "/pdca design test-qa-feature-01 -- Read the plan document and create a minimal design document." --yolo -o text 2>&1
# 확인:
ls docs/02-design/features/test-qa-feature-01.design.md 2>/dev/null
```

- **기대 결과**: Design 문서가 `docs/02-design/features/` 에 생성
- **통과 기준**: 파일 존재 + Plan 문서 참조
- **검증 포인트**: `design.template.md` 템플릿 구조 반영

### TC-P0-13: Context Anchor 전파

Design 문서에 Plan의 Context Anchor가 전파되었는지 확인.

```bash
grep -i "context\|anchor\|plan" docs/02-design/features/test-qa-feature-01.design.md 2>/dev/null
```

- **기대 결과**: Context Anchor 섹션 또는 Plan 참조가 Design에 존재
- **통과 기준**: Plan 관련 참조 텍스트 발견
- **검증 포인트**: PDCA 문서 간 traceability 확보

### TC-P0-14: PDCA Analyze (Gap Analysis)

Design과 구현 간 Gap Analysis 수행 기능 검증.

```bash
gemini -p "/pdca analyze test-qa-feature-01 -- Compare the design document against current implementation. Report match rate." --yolo -o text 2>&1
```

- **기대 결과**: Gap analysis 수행, match rate 백분율 보고
- **통과 기준**: match rate 퍼센트가 출력에 포함
- **검증 포인트**: `bkit.config.json`의 `matchRateThreshold: 90` 기준 적용

### TC-P0-15: PDCA Report 생성

PDCA cycle의 최종 Report 생성 기능 검증.

```bash
gemini -p "/pdca report test-qa-feature-01 -- Create a final report for this test feature." --yolo -o text 2>&1
```

- **기대 결과**: Report 문서 생성 또는 report 내용 출력
- **통과 기준**: Report 형식의 결과물 확인
- **검증 포인트**: `report.template.md` 템플릿 반영

## 18. Test Execution Guide

### 18.1 실행 방법

각 테스트를 수동으로 Gemini CLI에서 실행합니다.

```bash
# 단일 테스트 실행 + 결과 저장
gemini -p "<테스트 케이스의 prompt>" --yolo -o text 2>&1 | tee /tmp/tc-result.txt

# 통과 기준 확인 (수동 검증)
# 출력 내용이 기대 결과와 일치하는지 확인
```

### 18.2 Rate Limit 대응

- Gemini API는 rate limit 적용 (Free tier: 60 req/min)
- 테스트 간 2-3초 대기 권장
- 429 에러 발생 시 60초 대기 후 재시도
- P0 테스트를 먼저 실행하고, P1/P2/P3 순서로 진행

```bash
# Rate limit 대응 스크립트 예시
run_test() {
  echo "=== $1 ==="
  gemini -p "$2" --yolo -o text 2>&1 | head -20
  echo "---"
  sleep 3
}

run_test "TC-P0-01" "List extensions"
run_test "TC-P0-02" "What is your current working directory?"
# ...
```

### 18.3 결과 기록 형식

각 테스트 케이스에 대해 아래 형식으로 기록합니다.

| TC ID | Status | Actual Output (요약) | Notes |
|-------|--------|---------------------|-------|
| TC-P0-01 | PASS / FAIL / SKIP / ERROR | 출력 첫 3줄 요약 | 특이사항 |

Status 정의:
- **PASS**: 기대 결과와 일치
- **FAIL**: 기대 결과와 불일치 (버그)
- **SKIP**: 환경 문제 등으로 실행 불가
- **ERROR**: 테스트 실행 자체가 실패 (crash, timeout 등)

## 24. Appendix E: Known Issues & Limitations

### v2.0.4 알려진 제한사항

1. **Rate Limit**: Gemini API free tier 60 req/min 제한으로 대량 테스트 시 429 에러 발생 가능
2. **--yolo 모드**: 일부 보안 차단 테스트 (TC-P2-17)에서 --yolo 모드 동작이 일반 모드와 다를 수 있음
3. **Agent Memory**: 첫 세션에서는 agent memory가 비어있어 일부 응답 품질이 낮을 수 있음
4. **Context Window**: 매우 긴 프로젝트에서 context 압축 (PreCompress) 트리거 시점이 다를 수 있음
5. **Gemini CLI 버전**: v0.37.0에서 일부 hook API 변경 가능성 (v0.36.0 기준 테스트)

### v2.0.3 -> v2.0.4 변경 영향 범위

- `gemini-extension.json` version 업데이트
- `bkit.config.json` version 업데이트
- `hooks.json` description 업데이트
- GEMINI.md version 업데이트
- 기능 변경 없음 (버전 동기화 + 안정화)
