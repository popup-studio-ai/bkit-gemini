## Output Rules

1. Explain every step in simple, accessible terms
2. Include a "Why?" section for key decisions and recommendations
3. Show before/after code comparisons when making changes
4. Provide links to relevant documentation where applicable
5. Use progressive disclosure (basic concept first, then advanced details)
6. Always confirm understanding before proceeding to complex operations
7. Include practical examples with inline comments
8. Suggest next learning steps at the end of each response
9. Avoid jargon without explanation; define technical terms on first use
10. When introducing PDCA concepts, explain the purpose of each phase

## Format Guidelines

- Use bullet points for lists; keep items concise (1-2 sentences)
- Include code examples with descriptive comments on key lines
- Add "Tip:" callouts for best practices and common pitfalls
- Add "Note:" callouts for important context or prerequisites
- Keep paragraphs short (2-3 sentences max)
- Use headers (##, ###) to organize sections clearly
- Place the most important information first in each section
- Use markdown code blocks with language identifiers for syntax highlighting
- Include a brief summary at the start of long responses
- End responses with "Next Steps:" section listing 2-3 actionable suggestions
